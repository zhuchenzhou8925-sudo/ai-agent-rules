## Core Item Schema

每个检查项优先使用以下字段。精简输出可以隐藏非关键列，但内部判断不能丢失来源、风险和状态。

| 字段 | 规则 |
|---|---|
| 编号 | 使用稳定前缀和顺序号，如 `Q-001`、`S-001`、`T-001` |
| 一级分类 | 资格、商务、评分、技术、报价、签章、格式、密封、电子投标、递交等 |
| 二级分类 | 具体主题，如营业执照、类似业绩、实施计划、报价汇总 |
| 任务类型 | 编制、收集、确认、计算、签字、盖章、扫描、装订、密封、上传、递交、复核 |
| 检查事项 | 单一、可观察、可勾选的动作 |
| 招标要求 | 忠实摘要，不扩大原文后果 |
| 证明材料/交付物 | 需要准备或验证的证据 |
| 原文位置 | 文件名 + 章节/条款 + 页码；页码不可得时使用章节路径 + 关键词 |
| 响应位置 | 投标文件中应出现或实际出现的位置 |
| 复核方式 | 原文对照、交叉计算、证据核验、人工核验、系统核验等 |
| 责任人 | 未提供时写 `【待补充】`，不要猜测 |
| 内部截止时间 | 未提供时写 `【待补充】` |
| 状态 | 未开始、进行中、待复核、已完成、阻断、无需响应 |
| 风险等级 | 致命、高、中、低 |
| 风险类型 | 资格、符合性、评分、报价、合同、交付、格式、递交 |
| 缺失后果 | 使用来源支持的谨慎表述 |
| 备注 | 冲突、依赖、OCR 置信度或人工核验边界 |

对“招标文件 + 投标文件”对照任务，增加：

| 字段 | 可选值 |
|---|---|
| 覆盖结论 | 已覆盖、部分覆盖、未覆盖、冲突、无法验证 |
| 实际响应位置 | 投标文件文件名 + 章节 + 页码 |
| 问题描述 | 缺失、表述差异、证据不足或无法核验原因 |
| 修复动作 | 具体且不编造事实的修复建议 |

### Response Stage

- 投标响应：必须在文件中提供表格、承诺、方案或证明。
- 履约事项：中标后实施，投标阶段可能需要确认或承诺。
- 内部决策：需要业务、管理、法务或财务作出判断。
- 物理/系统操作：需要人员或真实系统完成并留痕。

## Risk Classification

| 等级 | 使用条件 | 典型示例 | 禁止做法 |
|---|---|---|---|
| 致命 | 原文明确为否决、无效、实质性条件，或明确硬性要求与直接否决后果有来源依据 | 资格不满足、超过最高限价、保证金不符、截止后递交 | 仅凭行业经验标记必然废标 |
| 高 | 可能造成重大扣分、关键响应缺失、严重合同或交付风险，但未有直接否决依据 | 高分评分材料缺失、关键技术条款未响应 | 把所有“须”一律标为致命 |
| 中 | 可能影响得分、完整性、履约或审查效率 | 证明链不完整、方案细节不足、一般偏差 | 对未核实后果作确定结论 |
| 低 | 主要影响规范性、可读性或低影响格式 | 目录、页码、命名、轻微排版 | 忽略明确格式否决条款；此时应升级 |

风险等级必须结合具体招标文件。同一事项在不同项目中可不同。

### Fatal-Risk Evidence Test

标为致命前逐项回答：

1. 来源是否明确写出否决、无效、实质性或直接后果？
2. 是否能定位到具体文件、条款或页码？
3. 是否存在其他章节的例外、修正规则或冲突？
4. 当前检查项是否准确对应原文主体、条件和时间范围？

任一关键答案为否时，降为高风险或标记 `【待确认】`，并说明需人工核对。

## Decomposition Rules

### Compound Qualification Clause

原文要求“提供法定代表人身份证明、授权委托书和代理人近三个月社保，并签字盖章”时，至少拆为：

- 收集法定代表人身份证明；
- 核对法定代表人身份证明有效性；
- 编制授权委托书；
- 核对法定代表人签字；
- 核对单位公章；
- 收集代理人身份证明；
- 收集代理人近三个月社保证明；
- 核对社保主体、人员和月份；
- 核对材料在资格章节的放置位置。

### Scoring Item

每个评分项至少拆出：

- 分值和评分公式；
- 满分所需数量或档位；
- 资格时间范围、金额、类型或人员条件；
- 每份证据的必要页面、签章和可识别字段；
- 材料表、正文和附件之间的一致性；
- 当前证据覆盖结论；
- 可能丢分点和补充动作。

### Technical Requirement

不得只写“完成技术方案”。按文件实际要求拆分整体架构、功能模块、设备与接口、数据与安全、实施阶段、进度里程碑、团队、培训、售后、质保、验收、交付物、驻场、演示、样品和知识产权等单元。

### Pricing

只生成复核项，不决定报价，也不提供底价公式、目标价格或报价策略。至少核对：最高限价、含税口径、税率、分项合计、总价、大小写、数量乘单价、暂列金额、折扣、签章、纸质/电子一致性和单独密封要求。成本构成、利润目标和最终价格由用户授权的财务及决策人员处理。

## Output Quality Rules

- 允许一条要求产生多个检查项；允许一个检查项引用多个来源。
- 合并重复项时保留全部有效来源和最严格但不冲突的条件。
- 发现冲突时不要“取最严格值”冒充招标结论；列出冲突并建议澄清。
- 状态“已完成”必须有材料、响应位置或人工/系统证据支持。
- Markdown 中使用 `- [ ]` 或表格状态列；Excel 中使用稳定字段和受控值，不用合并单元格承载关键数据。
- 长清单可以按章节拆分，但编号应稳定，便于追踪和复核。
