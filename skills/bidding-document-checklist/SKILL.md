# Bidding Document Checklist

## Purpose

将招标要求转化为最小可执行检查项，回答“做什么、依据是什么、放在哪里、如何复核、遗漏有什么风险”。保持每个高风险判断可追溯，不把一般要求夸大为否决投标条件。

## Related Skills

- 招标要求、评分逻辑、合同风险或响应策略仍需深度分析时，先使用 `bidding-document-analysis`，再将其结构化结果转换为 Checklist。
- 用户只调用本 Skill 时，直接完成最低限度的章节识别和要求提取，不强制用户先运行其他 Skill。
- 输出 Markdown 内容时直接生成；用户明确要求 `.xlsx`、`.docx`、`.pdf` 时，先完成内容与事实检查，再路由到对应文件能力。不要用本 Skill 实现排版引擎。
- 不用本 Skill 编写完整投标文件；正式响应文本另行路由到 `document-generation`，并保留招标合规人工复核。

## Inputs

接受单个完整文件、多个分册、章节摘录或“招标文件 + 已编制投标文件”。先识别可用材料：

- 招标公告或邀请书；
- 投标人须知及前附表；
- 资格条件、否决投标条款和实质性要求；
- 评标办法；
- 技术需求或任务书；
- 商务与合同条款；
- 投标文件组成与格式；
- 报价、签章、装订、密封、电子上传、递交和开标要求；
- 已提供的企业资质、业绩、人员、证明材料或投标文件草稿。

默认只读源文件。处理文件夹时先列清单、识别版本和缺失章节；不要移动、重命名、覆盖或归档原文件。

## Output Set

默认生成：

1. 项目基本信息与材料完整性说明；
2. 致命风险清单；
3. 投标文件编制 Checklist；
4. 评分项满分覆盖 Checklist；
5. 投标文件终审 Checklist；
6. 待补充材料与待确认事项；
7. 来源冲突和人工复核说明。

用户指定精简范围时只输出需要的视图。字段、分类和模板见 [Checklist Schema and Risk Rules](references/checklist-schema-and-risk-rules.md)。需要验证或改进本 Skill 时，读取 [Validation Scenarios](references/validation-scenarios.md)。

## Execution Flow

### 1. Inventory And Completeness

列出文件、分册、版本、页数或可识别章节。把缺失分为：

- 完全阻断：文件无法打开、加密、严重缺页、正文不可识别，或只有空白模板而无任何招标要求；
- 部分阻断：缺少评标办法、技术需求、合同、投标格式或递交规则。

部分阻断时继续生成可生成部分，并明确受影响视图。扫描件 OCR 质量较差时标注识别置信度和需人工核对的位置。

### 2. Build A Requirement Ledger

按章节提取项目基本信息、资格、实质性要求、评分、技术、商务合同、报价、格式签章和递交要求。优先搜索“必须、须、应当、不得、不接受、否决、无效、实质性、★、最高限价、截止”等表达，但不能只依赖关键词。

为每条要求保留：文件名、章节或条款号、页码（可得时）、原文摘要。页码不可得时至少保留章节路径和检索关键词。不要伪造页码或条款号。

### 3. Resolve Facts And Conflicts

对同一事实跨公告、须知、评分办法、合同、技术需求和格式模板进行一致性检查。不要自行选择冲突版本：

- 完全缺失的必需信息写 `【待补充】`；
- 已有候选值但冲突、含糊、推断或未核实写 `【待确认】`；
- 明确冲突同时列出各来源、影响和建议澄清动作。

不得把示例数据、历史项目材料或其他主体资质当作当前投标事实。

### 4. Atomize Requirements

把复合条款拆为可单独完成和复核的动作。每项只保留一个主要动作，例如“收集授权委托书”“核对法定代表人签字”“核对单位公章”“核对代理人社保月份”。

为每项指定任务类型：编制、收集、确认、计算、签字、盖章、扫描、装订、密封、上传、递交或复核。不得只生成“完成技术方案”“检查资格材料”等不可验证的大项。

### 5. Map Evidence And Response Location

建立链路：

`招标要求 -> 响应动作 -> 证明材料/交付物 -> 投标章节 -> 复核方法`

资格项拆出主体、材料、时间范围、数量、有效期、原件/复印件、签章和放置位置。评分项拆出分值、规则、满分条件、证据组合、数量上限和可能丢分点。技术需求拆到功能、参数、接口、安全、实施、人员、培训、售后、验收和交付物等最小可检查单元。

### 6. Classify Risk

使用致命、高、中、低四级和资格、符合性、评分、报价、合同、交付、格式、递交等风险类型。

只有在来源明确列为否决条件、无效条件、实质性要求，或明确硬性条件与直接后果有文本依据时，才标为“致命”。证据不足时不得根据常识声称必然废标，改标为高风险或 `【待确认】`。统一采用谨慎措辞：“缺失可能导致投标被否决，最终以招标文件和评标认定为准。”

### 7. Generate Workflow And Final-Review Views

至少按以下顺序组织编制视图：项目基本信息、资格、授权、商务、评分、技术、报价、偏离与承诺、签章、格式、电子文件、装订密封、递交开标。

终审视图聚焦可在提交前验证的结果。区分：

- 投标响应：必须在投标文件中体现；
- 履约事项：中标后执行，但投标阶段可能需承诺；
- 内部决策：付款、责任、知识产权、驻场等需业务确认；
- 物理或系统操作：签章原件、密封、上传、递交等必须由人员或实际系统复核，不能仅凭文本断言完成。

### 8. Compare Draft When Provided

同时提供投标文件时，逐项标记：已覆盖、部分覆盖、未覆盖、冲突、无法验证。记录实际响应位置和证据，不因出现相似关键词就判定已覆盖。签字、印章真伪、密封状态、上传成功与否只能标为待人工或系统核验。

### 9. Quality Check

交付前检查：

- 所有致命和高风险项均有来源；
- 评分办法逐项覆盖，分值和满分条件可复核；
- 复合条款已拆分，重复项已合并但来源未丢失；
- 招标公告、须知、技术需求、合同和格式之间的冲突已列出；
- 缺失事实和冲突事实分别使用正确占位符；
- 未编造企业材料、人员、业绩、报价、签章或完成状态；
- Checklist 可直接用于责任分配、状态跟踪和终审；
- 明确提示仍需招标、法务、财务、授权人员或封标人员复核的事项。

## Safety Boundaries

不得：

- 自动决定、计算或承诺投标报价，或提供底价公式、目标价格和报价策略；本 Skill 只核对招标口径、最高限价、必填字段和算术/版本一致性；
- 编造资质、业绩、人员、证书、社保、财务数据、签字、印章或授权关系；
- 提供保证有效、保证中标或正式法律意见；
- 修改招标文件、投标文件或源材料；
- 代替人员完成签字盖章、物理密封、系统上传或现场递交；
- 将真实客户、合同、报价、招采或企业敏感材料发布到 GitHub 或外部渠道。

若用户要求伪造材料、虚假承诺或绕过招标要求，拒绝该部分，但继续提供合规的缺口清单和补充建议。

## Acceptance Criteria

仅当以下条件满足时称为“可用于编制与终审”，不得称为“保证合规”或“最终提交就绪”：

- 输入完整性和局部阻断已说明；
- 明确否决条件、资格条件、评分项和递交要求已覆盖；
- 技术、商务合同、报价、格式签章和电子/纸质要求按现有材料覆盖；
- 每个致命或高风险项可追溯；
- 检查项原子化且具有动作、材料、位置、风险和复核方法；
- 缺失、冲突、推断和物理核验边界可见；
- 输出未越界到报价决策、材料编造、法律结论或自动提交。

## Response Shape

先给结论和材料完整性，再给清单。表格较长时优先保持字段稳定，不用长段落替代检查项。末尾只列：致命未闭环项、`【待补充】`、`【待确认】`、人工/系统复核项和下一步。
