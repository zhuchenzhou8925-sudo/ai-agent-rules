# Knowledge Management Acceptance Checklist

Use this checklist before treating a knowledge-management run as ready, especially before formal KnowledgeBase filing or GitHub-oriented handoff.

## Trigger Boundary

- [ ] The user's primary goal is knowledge-base sedimentation, material archiving, Obsidian / Markdown knowledge organization, resource indexing, topic notes, MOCs, knowledge cards, backlinks, tags, or frontmatter links.
- [ ] The task is not primarily formal bidding, tender, procurement, scoring, response, or contract-risk analysis. If it is, `bidding-document-analysis` was used first.
- [ ] The task is not only a one-time Word, PPT, PDF, Excel, report, proposal, or presentation output unless the user explicitly asked to file it into the knowledge base.
- [ ] For mixed business materials, the main goal was clarified before choosing the workflow.

## Read And Classify

- [ ] Source scope was authorized and not expanded silently.
- [ ] Local-file work began with read-only scanning and inventory.
- [ ] The material type was identified before deciding the artifact shape.
- [ ] The recommended topic note type was selected.
- [ ] The need for a resource index, topic note, knowledge card set, MOC, or combination was checked.

## Merge, Link, And File

- [ ] Duplicate, merge, de-duplication, version update, and related-note conflicts were checked.
- [ ] MOC creation, MOC attachment, or upper-level MOC update was checked.
- [ ] Formal KnowledgeBase write authorization was confirmed before creating or updating formal artifacts.
- [ ] Resource indexes, topic notes, knowledge cards, or MOCs were generated or updated only when appropriate.
- [ ] Source traceability was preserved with `source` or `source_path`.

## Metadata And Quality

- [ ] Frontmatter includes required fields such as `created`, and any relevant `reviewed` or `status` field was checked.
- [ ] Backlinks, frontmatter `links`, and tags were checked.
- [ ] The output contains business understanding, structural analysis, mechanism breakdown, applicable boundaries, related knowledge links, and reusable wording when relevant.
- [ ] The result was checked for shallow summaries, raw material stacking, weak reuse value, unclear boundaries, and missing related-knowledge links.

## Storage And Safety

- [ ] Process checks, validation records, logs, previews, exception reports, backups, and temporary outputs were kept out of the formal KnowledgeBase.
- [ ] `<CODEX_OUTPUT_DIR>` was avoided unless the output was a test, preview, exception report, process check, backup, or explicitly requested temporary output.
- [ ] Private vault content, customer materials, contracts, quotations, bidding files, run logs, secrets, tokens, keys, and unredacted business materials were not synced to GitHub.
- [ ] No original source files were moved, deleted, renamed, overwritten, or archived without explicit user confirmation.
- [ ] Any blocker, missing dependency, unreadable source, unclear target, permission issue, duplicate conflict, or sensitive-content concern was reported.
- [ ] Confirmation nodes were respected before formal filing, destructive actions, Git operations, publication, or scope expansion.
