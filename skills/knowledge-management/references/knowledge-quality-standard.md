# knowledge-quality-standard

## 1. 文件定位

本文件定义 Obsidian 正式知识化入档的统一知识质量标准。它约束资料索引、MOC、主题笔记、项目笔记、资料笔记、知识卡片和未来扩展知识成果。

知识资产不是资料摘要，而是能够支持方案、PPT、投标、汇报、项目复盘、产品设计、技术选型、商务谈判和知识复用的结构化成果。

## 2. 适用范围

适用于以下任务：

- 项目资料、可研报告、研究报告、文献资料、技术文件、产品资料、政策标准、商务资料、培训资料、方法论资料和混合资料的知识化入档；
- 正式 KnowledgeBase 写入前后的质量检查；
- 主题笔记、知识卡片、MOC、资料索引的质量判断；
- 后续新增资料类型和笔记类型的最低质量约束。

不适用于过程日志、异常报告、测试输出、临时预览、运行记录和未确认中间成果。

## 3. 与正式知识化入档的关系

- 用户授权资料默认视为有知识化价值；Codex 不反复判断“是否值得入库”，而是判断“如何高质量入库”。
- 入库前必须同时判断资料类型、主题笔记类型、知识卡片类型、MOC 新建或挂接、合并去重、版本更新和质量检查。
- 未达到本文件最低合格标准的成果不得标记为正式高质量入库成果。
- 质量不足但来源重要的内容，应记录待补充项、边界和更新条件，不得把缺口伪装成确定结论。

## 4. KQ-12 知识资产质量标准

| 编号 | 维度 | 要求 |
|---|---|---|
| KQ-01 | 准确性 | 事实、数据、政策、项目表述、产品能力和商务信息必须来自可追溯来源；不能把未核验内容写成确定事实。 |
| KQ-02 | 完整性 | 应覆盖背景、对象、问题、结论、依据、边界、关联和后续动作；不同资料类型按对应模板补足关键章节。 |
| KQ-03 | 证据性 | 关键判断必须说明依据来源；数据、政策、合同、价格、技术指标和案例效果应标注来源或待核验。 |
| KQ-04 | 结构性 | 必须按资料类型使用稳定结构，避免把原文目录机械堆叠为正文。 |
| KQ-05 | 一致性 | 命名、frontmatter、标签、双链、MOC 挂接、状态字段和来源字段必须一致。 |
| KQ-06 | 专业性 | 表述应符合建筑业、BIM、数字化交付、智能建造、智慧工地、项目管理、产品和商务语境。 |
| KQ-07 | 可复用性 | 必须说明可用于方案、PPT、投标、汇报、复盘、产品设计、技术选型、商务谈判或客户沟通的方式。 |
| KQ-08 | 可追溯性 | 应能从 MOC、主题笔记、知识卡片、资料索引回到原始资料或来源说明。 |
| KQ-09 | 非重复性 | 入库前必须检查同主题笔记、同主题卡片、同主题 MOC 和版本更新；重复则合并、更新或挂接，不新建近似成果。 |
| KQ-10 | 可维护性 | 应标注状态、更新时间、待补充项、更新条件、版本关系和需要人工确认的事项。 |
| KQ-11 | 可扩展性 | 新资料类型可新增扩展笔记类型，但必须定义适用资料、必备结构、质量标准、关联方式和质检要求。 |
| KQ-12 | 可行动性 | 成果应能支持后续实际工作输出，不能只形成概念摘录、浅摘要或空泛结论。 |

## 4.1 知识内容深度合格标准

正式入档闭环通过不等于知识质量通过。

frontmatter、标签、双链、MOC 挂接、资料索引和 status 只代表结构闭环；正式知识成果还必须单独满足内容深度要求。

合格的知识内容至少应体现：

- 原始资料的核心问题、对象、场景、目标和约束；
- 原始资料中的章节结构、表格、清单、技术路线、功能模块、实施路径、风险和效益等有效信息；
- 从资料事实到专业判断的依据链条；
- 可用于方案、PPT、投标、汇报、复盘、产品设计、技术选型或商务沟通的复用表达；
- 适用边界、前提条件、反例或不适用场景；
- 与主题笔记、知识卡片、MOC、资料索引之间的可追溯关系。

以下情况应判定为 P2 知识质量问题：

- 主题笔记只是摘要或目录复述，缺少问题、判断、路径、依据、复用价值和边界；
- 知识卡片只是概念说明或价值描述，缺少判断条件、方法框架、使用方式、反例和更新条件；
- 原始资料有表格、功能清单、技术路线、实施计划、风险清单、指标或对比内容，但入库成果未结构化利用；
- 只完成了命名、frontmatter、标签、双链和 MOC 挂接，但内容不足以支撑后续工作输出。

笔记篇幅只能作为风险信号，不能作为唯一质量标准。内容较短但判断完整、证据充分、可复用性明确，可以合格；内容较长但缺少判断链和复用结构，仍不合格。

## 4.2 正式成果增强修正规则

当正式知识成果已经入库，但检查发现内容深度不足时，应进入“正式成果增强修正”，而不是重新走草稿、迁移候选或临时中转流程。

正式成果增强修正必须遵守：

- 重新读取原始资料或已授权来源；
- 对比现有正式成果，保留正确的 frontmatter、status、source、source_files、tags、links、related_moc 和已有有效内容；
- 不删除、覆盖或移动无关正式知识文件；
- 不写入 `_Codex_Output`，除非用户明确要求临时输出或规则允许测试、预览、异常报告、过程检查、备份；
- 针对主题笔记补强背景、问题、论证链条、结构化拆解、专业判断、复用表达和边界；
- 针对知识卡片补强核心结论、卡片类型、适用场景、判断条件、方法框架、证据来源、使用方式、反例、关联知识和更新条件；
- 仅在具备跨项目、跨场景复用价值时新增知识卡片；
- 所有新增判断必须可追溯到原始资料、官方来源或已入库资料；
- 修正后重新执行统一质检，并输出修改说明。

正式成果增强修正属于 formal knowledge filing 的质量补强动作，不得恢复 draft / migration 默认流程。

## 5. 事实、判断、推断、建议区分规则

| 类型 | 定义 | 写法要求 |
|---|---|---|
| 事实 | 来源资料明确给出的项目、时间、主体、数据、条款、功能、价格、范围等信息 | 写明来源；正式引用前需核验的数据应标注“来自来源资料，正式引用前建议核验”。 |
| 判断 | Codex 基于资料和专业语境形成的结论 | 必须说明依据和适用边界，不得伪装为原文事实。 |
| 推断 | 来源未直接说明，但可由上下文合理推得的内容 | 必须标注“推断”或放入分析备注，不得写成确定事实。 |
| 建议 | 面向后续方案、产品、技术、商务或管理动作的建议 | 必须说明前提、适用条件、风险和需要用户确认的事项。 |

## 6. 不确定信息处理规则

- 来源缺失、版本不清、数据口径不明、政策时效不明、合同法律性质不明、产品最新状态不明时，必须标注待确认。
- 不确定内容不得进入“核心结论”作为确定判断，可进入“待确认事项”“风险边界”“分析备注”或 frontmatter 的 review/process 字段。
- 涉及法律性质结论、合同责任、索赔、违约、合规处罚等内容时，必须标注“需法务确认”。
- 涉及正式对外承诺、商务合作模式、产品最新能力、敏感客户案例时，必须交由用户确认。

## 7. 反浅层规则

禁止以下低质量成果：

- 只摘录目录、摘要、关键词，没有核心判断和业务启发；
- 只改写原文段落，没有结构重组、机制拆解和适用边界；
- 使用“赋能、闭环、抓手、底座、生态”等词堆砌，缺少具体对象、动作、依据和场景；
- 把项目阶段性事实直接泛化为通用规律；
- 把政策鼓励性表述写成强制要求；
- 把宣传话术写成已验证产品能力；
- 把合同、报价、招采内容抽象成通用策略时不说明边界。

## 8. 与知识成果类型的关系

- 资料索引：负责说明来源、分类、版本关系、资料关系、核心判断、业务启发和后续沉淀方向。
- 主题笔记：负责承载某类资料的完整分析主线，按资料类型使用对应模板。
- 项目笔记：只负责项目、客户、任务、交付、方案、实施等项目型资料，不替代全部主题笔记。
- 知识卡片：只沉淀跨项目、跨场景可复用的单一知识点，不承担完整资料背景。
- MOC：负责主题导航、挂接关系和后续补充方向，不写成完整报告。

## 9. 最低合格标准

正式知识成果至少必须满足：

- 来源明确，可追溯到原始资料、资料索引、主题笔记或用户输入；
- 已识别主资料类型和必要的次资料类型；
- 已判断主题笔记类型、知识卡片类型、MOC 新建/挂接、合并/去重、版本更新；
- 正文包含核心事实、关键判断、依据、适用边界、关联知识和可复用方向；
- frontmatter、状态、标签、双链和来源字段完整；
- 不含 Codex 执行过程、入档建议、检查过程等流程污染；
- 通过 `quality-checklist.md` 的入库前检查。
