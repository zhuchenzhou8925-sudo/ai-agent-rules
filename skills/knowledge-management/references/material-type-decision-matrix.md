# material-type-decision-matrix

## 1. 文件定位

本文件用于 Obsidian 知识化入档前的资料类型识别。它决定主题笔记类型、知识卡片类型、MOC 处理方式、合并去重方式、版本更新方式和人工确认事项。

用户授权资料默认有知识化价值。Codex 不判断“是否值得入库”，而是判断“如何入库”。

## 2. 识别顺序

1. 识别主资料类型：决定主题笔记类型和主要质量标准。
2. 识别次资料类型：决定补充章节、知识卡片类型和风险检查。
3. 判断资料关系：同一项目多文件、同一主题多来源、同一资料不同版本、附件/正文关系、政策/解读关系、技术/产品/商务混合关系。
4. 判断是否已有同主题 MOC、主题笔记、知识卡片或资料索引。
5. 输出资料类型识别表。

## 3. 主资料类型识别规则

| 主资料类型 | 典型资料 | 识别特征 | 建议主题笔记类型 |
|---|---|---|---|
| 项目资料 | 项目方案、实施材料、会议纪要、交付材料、客户需求、项目报价说明 | 有具体项目、客户、任务、交付、实施路径、阶段结果或项目风险 | 项目笔记 |
| 可研报告 / 研究报告 | 可研、专题研究、行业研究、市场分析 | 有研究对象、研究问题、论证结构、数据依据、结论建议 | 研究笔记 |
| 文献资料 | 论文、学术文章、白皮书、理论资料 | 有作者/机构、研究问题、模型方法、证据、局限性 | 文献笔记 |
| 技术文件 | 技术方案、架构设计、接口文档、部署说明、开发说明 | 有架构、模块、接口、数据、流程、依赖、运行条件 | 技术笔记 |
| 产品宣传 / 产品手册 / 竞品资料 | 产品手册、官网介绍、宣传册、竞品资料、销售材料 | 有目标用户、场景、功能能力、卖点、差异化或宣传话术 | 产品笔记 |
| 政策标准 / 指南 / 规范 | 政策、标准、指南、管理办法、规范 | 有发布主体、适用范围、条款要求、必须/应/宜/可等约束词 | 政策标准笔记 |
| 商务资料 / 报价 / 合同 / 招采文件 | 报价单、合同、招标文件、采购需求、谈判材料 | 有交易对象、范围、价格、付款、交付、验收、责任、风险 | 商务笔记 |
| 培训资料 / 方法论 / 操作手册 | 培训课件、制度流程、SOP、操作手册、工作流 | 有适用对象、流程步骤、输入输出、质量标准、常见问题 | 方法笔记 |
| 混合资料 | 多类型资料包、项目+技术+商务、政策+标准+解读、可研+附件+数据表 | 同时满足多个类型，且单一类型不能完整解释资料价值 | 主类型对应主题笔记，次类型进入补充章节 |
| 未来扩展资料类型 | 新行业、新对象、新知识形态 | 无法归入现有类型，且有稳定复用场景 | 扩展笔记类型 |

## 4. 次资料类型识别规则

- 次资料类型不决定主笔记名称，但必须影响补充章节和卡片分型。
- 项目资料中含报价、合同、采购需求时，次类型为商务资料，必须补充范围、价格、交付、验收、责任和风险。
- 项目资料中含架构、接口、系统流程时，次类型为技术文件，必须补充技术路径、依赖和风险。
- 产品资料中含竞品对照时，次类型为对比资料，优先生成对比卡片。
- 政策资料中含解读、案例或地方执行口径时，次类型为研究/方法资料，必须区分原文要求和解读判断。

## 5. 混合资料处理规则

| 混合关系 | 处理方式 |
|---|---|
| 同一项目多文件 | 先生成资料索引，主笔记为项目笔记，按版本和附件关系组织。 |
| 同一主题多来源 | 先生成资料索引或 MOC，主笔记按主题主类型确定，来源之间做互证和冲突标注。 |
| 同一资料不同版本 | 优先更新或合并已有笔记，标注主版本、历史版本、变更点，不重复新建。 |
| 技术 + 产品 + 商务 | 主类型按用户目标确定；技术进技术章节，产品进能力章节，商务进范围/风险章节。 |
| 政策 + 标准 + 解读 | 政策标准笔记为主，严格区分原文要求、标准条款、解读判断和业务影响。 |
| 可研 + 附件 + 数据表 | 研究笔记为主，附件和数据表进入证据、口径、数据依据和待核验章节。 |

## 6. 知识卡片生成判断

生成知识卡片的条件：

- 有跨项目、跨场景复用价值；
- 有稳定方法、模型、判断逻辑、要求、风险、表达或对比价值；
- 来源明确，可追溯；
- 能独立说明一个知识点，且不依赖完整项目背景才能理解。

不生成知识卡片的条件：

- 仅是一次性事实、临时任务、版本未确认信息；
- 已有同主题卡片可更新；
- 信息不足以支撑可复用判断；
- 只能作为主题笔记背景或附件说明。

## 7. 卡片类型建议

| 来源特征 | 建议卡片类型 |
|---|---|
| 判断标准、选择条件、适用性判断 | 判断卡片 |
| 操作步骤、工作流程、实施路径 | 方法卡片 |
| 分析框架、能力模型、架构模型 | 模型卡片 |
| 应用场景、客户场景、业务场景 | 场景卡片 |
| 政策条款、合同要求、交付要求、验收要求 | 要求卡片 |
| 合规风险、实施风险、商务风险、技术风险 | 风险卡片 |
| 汇报话术、方案表述、投标表达 | 表达卡片 |
| 产品差异、版本差异、方案差异、竞品差异 | 对比卡片 |
| 新知识形态 | 未来扩展卡片 |

## 8. MOC、合并、版本和人工确认

- 新建 MOC：主题长期复用、已有 2 个以上相关成果、资料组需要持续补充、无合适既有 MOC。
- 挂接已有 MOC：已有同主题或上层主题 MOC，且新成果能自然纳入。
- 合并已有笔记：同主题、同项目、同研究对象、同政策文件、同产品或同版本链条已经存在。
- 版本更新：同文件、同项目、同政策、同产品资料出现新版本、修订版、补充附件或口径更新。
- 需要人工确认：目标正式目录不明确、版本边界影响结论、法律/商务/对外承诺/产品最新状态不明确、同名冲突、敏感内容。
