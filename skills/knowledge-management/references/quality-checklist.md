# quality-checklist

## 1. 文件定位

本文件是 Obsidian 正式知识化入档的统一质量检查矩阵。它用于入库前、入库后和规则测试阶段的简短质检，不替代 `knowledge-quality-standard.md`、`topic-note-quality-standard.md` 或 `knowledge-card-quality-standard.md`。

## 2. 入库前检查

| 检查项 | 要求 |
|---|---|
| 授权范围 | 只读取用户授权资料，不扩大范围。 |
| 资料类型识别 | 已按 `material-type-decision-matrix.md` 判断主类型、次类型和混合关系。 |
| 重复/合并 | 已检查同主题 MOC、主题笔记、资料索引、知识卡片和版本更新。 |
| 输出类型 | 已判断主题笔记类型、知识卡片类型、MOC 新建/挂接和资料索引需求。 |
| 来源追踪 | 来源路径、来源说明或 URL 可追溯。 |
| 不确定事项 | 待确认、版本不清、法律/商务/产品最新状态等已标注。 |

## 3. 入库后检查

| 检查项 | 要求 |
|---|---|
| 正式状态 | 正式成果状态为 `reviewed`。 |
| 正式目录 | 成果进入正确 KnowledgeBase 正式分类。 |
| MOC 挂接 | 已挂接本次或已有 MOC；暂不挂接需说明原因。 |
| 临时输出清理 | 如使用 `_Codex_Output`，按用户授权清理对应临时输出。 |
| 边界确认 | 原始资料未修改、未移动、未删除、未重命名。 |

## 4. 主题笔记检查

| 检查项 | 要求 |
|---|---|
| 笔记类型 | 项目、研究、文献、技术、产品、政策标准、商务、方法或扩展类型明确。 |
| 结构 | 满足 `topic-note-quality-standard.md` 对应模板。 |
| 结论 | 有核心判断，不停留在摘要。 |
| 依据 | 关键判断有来源、数据、条款或资料依据。 |
| 边界 | 有适用场景、不适用场景、风险或前提。 |
| 复用 | 明确可用于方案、可研、汇报、投标、产品、技术或商务工作。 |

## 5. 知识卡片检查

| 检查项 | 要求 |
|---|---|
| 是否必要 | 不是所有资料都强制生成卡片；无复用价值则不生成。 |
| 卡片类型 | 判断、方法、模型、场景、要求、风险、表达、对比或扩展类型明确。 |
| 来源 | 可追溯到资料、主题笔记、资料索引或用户输入。 |
| 复用 | 有跨项目、跨场景复用价值。 |
| 边界 | 有适用边界、反例或使用限制。 |
| 重复 | 已检查同主题卡片，重复则更新或合并。 |

## 6. MOC 挂接检查

| 检查项 | 要求 |
|---|---|
| 是否需要 MOC | 主题长期复用、成果多、资料组持续补充时应新建或挂接 MOC。 |
| 既有 MOC | 已检查同主题或上层 MOC。 |
| 双链 | MOC 链接资料索引、主题笔记、知识卡片。 |
| 后续补充 | 未生成但应补充的内容放入后续补充，不链接不存在文件为已生成成果。 |

## 7. frontmatter 检查

| 检查项 | 要求 |
|---|---|
| title | 与文件主题一致。 |
| type | 与成果类型一致。 |
| status | 正式入库为 `reviewed`；测试、预览或临时输出不得标记为正式入库状态。 |
| source/source_path | 来源清楚。 |
| tags | 包含内容类型、主题领域和状态标签。 |
| links | 包含相关 MOC、主题笔记、资料索引或知识卡片。 |
| review | 记录内容检查、来源追踪和正式入库状态。 |

## 8. 双链与标签检查

- 双链使用 `[[页面名]]`，不写 `.md` 后缀。
- 不把不存在文件链接为已生成成果。
- 标签使用中文业务标签，避免同义标签散乱。
- 标签至少覆盖内容类型、主题领域、状态。

## 9. 重复/合并检查

- 同项目、同主题、同资料、同政策、同产品、同卡片类型必须检查重复。
- 新版本优先更新或合并旧笔记，不默认新建。
- 冲突或无法判断时，停止在确认节点，不绕过 blocker。

## 10. 证据性、可复用性和风险检查

| 检查项 | 要求 |
|---|---|
| 证据性 | 关键判断、数据、条款、价格、技术指标有来源或待确认标注。 |
| 可复用性 | 明确复用对象和使用方式。 |
| 风险 | 技术、商务、合规、实施、数据和版本风险已标注。 |
| 不确定性 | 推断、建议、待核验内容不写成确定事实。 |

## 10.1 内容深度质检项

结构闭环通过后，必须继续检查内容深度。

| 检查项 | 合格标准 | 结果 |
|---|---|---|
| 原文结构利用 | 是否使用原文中的章节、表格、清单、技术路线、功能模块、实施计划、风险、指标或对比内容 | |
| 背景与问题 | 是否说明资料解决的问题、对象、场景和约束 | |
| 判断链条 | 是否从资料事实推导出专业判断，并说明依据 | |
| 结构化拆解 | 是否拆解需求、能力、模块、流程、数据对象、角色、风险、价值等关键内容 | |
| 可复用表达 | 是否沉淀可用于方案、PPT、投标、汇报、产品设计、技术选型或复盘的表达 | |
| 边界与反例 | 是否说明适用条件、不适用场景、反例或待补充项 | |
| 知识卡片工具性 | 卡片是否具备判断条件、方法框架、检查清单、使用方式和更新条件 | |

任一关键项缺失，应判定为“需增强”。若该成果已经正式入库，应进入正式成果增强修正。

## 10.2 正式成果增强修正检查

正式成果增强修正应检查：

- 是否重新读取原始资料或已授权来源；
- 是否保留正确 frontmatter、status、source、source_files、tags、links、related_moc 和已有有效内容；
- 是否未写入 `_Codex_Output`；
- 是否未生成草稿、迁移候选或临时中转成果；
- 是否未删除、覆盖或移动无关正式知识文件；
- 是否补强主题笔记的背景、问题、论证链条、结构化拆解、专业判断、复用表达和边界；
- 是否补强知识卡片的核心结论、卡片类型、判断条件、方法框架、证据来源、使用方式、反例、关联知识和更新条件；
- 是否只在具备跨项目、跨场景复用价值时新增知识卡片；
- 是否重新执行本质检清单。

## 11. 过程信息隔离检查

正式知识正文不得写入：

- Codex 执行过程；
- 入档建议；
- 是否等待用户确认；
- 原文件是否移动；
- 临时检查过程；
- 运行日志；
- 异常处理过程。

上述内容如必须保留，只能进入 frontmatter、`_process` 元数据区、最终短报告或 Codex_Control 运行记录。

## 12. 简短质检结果格式

每次入库后可输出：

| 检查项 | 结果 | 说明 |
|---|---|---|
| 资料类型识别 | 通过 / 需修正 |  |
| 主题笔记质量 | 通过 / 需修正 |  |
| 知识卡片质量 | 通过 / 不适用 / 需修正 |  |
| MOC 挂接 | 通过 / 需补充 |  |
| 重复检查 | 未发现 / 已合并 / 需人工确认 |  |
| 证据与来源 | 通过 / 需补充 |  |
| 可复用性 | 高 / 中 / 低 / 不适用 |  |
| 内容深度 | 通过 / 需增强 |  |
| 原文结构利用 | 通过 / 需增强 |  |
| 增强修正 | 不需要 / 已完成 / 需执行 |  |
| 风险项 | 无 / 有 |  |
| 正式入档状态 | 通过 / 需确认 |  |
