# bidding-document-analysis acceptance checklist

本文件用于验收 `bidding-document-analysis` 是否能稳定完成投标文件分析 MVP。

本清单只用于验收 `bidding-document-analysis` 的工作流能力，不替代正式投标评审、法律审查或报价决策。通过本清单只能说明该 Skill 能完成结构化分析、边界控制和测试输出，不代表投标文件已经可以直接提交。

## 基础触发检查

- [ ] 是否能在用户要求分析招标文件、采购文件、评分办法、合同条款或响应要求时触发。
- [ ] 是否能识别当前任务是投标文件分析，而不是正式投标文件编制。
- [ ] 是否能明确第一版目标是 MVP 分析，不承诺一次性生成成熟投标工作流。
- [ ] 是否能在输出前说明本次分析的目标、范围和限制。

## 文件读取边界检查

- [ ] 是否默认采用只读分析。
- [ ] 是否在读取大批量文件前确认读取范围。
- [ ] 是否未修改、移动、删除、重命名、覆盖、归档或提交任何原始文件。
- [ ] 是否能列出已读取文件和未读取文件。
- [ ] 是否能标注文件版本、来源或权限不明确的问题。

## 项目基本信息提取检查

- [ ] 是否提取项目名称、采购人、招标人或代理机构。
- [ ] 是否提取采购范围、服务内容、建设地点或实施范围。
- [ ] 是否提取投标截止时间、开标时间、递交方式等关键时间要求。
- [ ] 是否提取服务周期、交付周期、质保或运维周期。
- [ ] 是否标注无法确认或资料缺失的信息。

## 采购需求拆解检查

- [ ] 是否拆解采购需求、建设内容、服务范围和技术要求。
- [ ] 是否区分必须响应项、建议响应项和可选补充项。
- [ ] 是否识别交付物、系统功能、服务内容、实施要求和验收要求。
- [ ] 是否将需求拆解为可用于投标响应的条目。
- [ ] 是否标注需求不清、前后矛盾或缺少附件的情况。

## 资格条件识别检查

- [ ] 是否提取供应商资格、资质、业绩、人员、财务、信誉等要求。
- [ ] 是否区分资格审查条件、评分加分条件和响应材料要求。
- [ ] 是否识别联合体、分包、授权、证明文件等特殊要求。
- [ ] 是否列出需要用户补充确认的资质或证明材料。

## 实质性响应要求检查

- [ ] 是否识别实质性响应条款、强制性条款和不可偏离条款。
- [ ] 是否提取报价、工期、质量、服务、验收、付款、合同等响应要求。
- [ ] 是否将实质性响应要求转化为可检查清单。
- [ ] 是否标注不响应、漏响应或偏离可能造成的风险。

## 废标风险识别检查

- [ ] 是否提取废标、无效投标、否决投标或不予受理条款。
- [ ] 是否识别文件格式、签章、授权、报价、递交、资格、响应偏离等风险。
- [ ] 是否区分高风险、需确认和一般注意事项。
- [ ] 是否说明每项风险对应的触发条件。
- [ ] 是否在用户要求跳过风险分析时停止并标记 blocker。

## 评分项提取检查

- [ ] 是否提取评分办法、分值构成和评审维度。
- [ ] 是否区分商务、技术、价格、服务、业绩、人员、方案等评分项。
- [ ] 是否识别客观评分项和主观评分项。
- [ ] 是否提取每项评分对应的证明材料、响应重点和失分风险。
- [ ] 是否能形成评分项提取表。

## 合同条款风险识别检查

- [ ] 是否识别付款、验收、违约责任、赔偿责任、知识产权、保密、变更、解除、争议解决等条款。
- [ ] 是否从乙方或投标响应方视角识别业务风险。
- [ ] 是否区分合同风险提示和法律意见。
- [ ] 是否明确输出不构成法律意见，需由用户或法律专业人员复核。
- [ ] 是否标注合同条款缺失、版本不明或前后冲突的情况。

## 交付要求梳理检查

- [ ] 是否提取交付内容、交付节点、交付形式和交付验收标准。
- [ ] 是否识别培训、运维、质保、售后、驻场、汇报等服务要求。
- [ ] 是否识别项目管理、进度计划、质量控制和验收配合要求。
- [ ] 是否将交付要求转化为投标方案编制要点。

## 响应文件框架检查

- [ ] 是否根据采购需求和评分办法生成响应文件编制框架。
- [ ] 是否区分商务部分、技术部分、报价部分和附件证明材料。
- [ ] 是否将评分项映射到响应章节或材料位置。
- [ ] 是否明确该框架是编制建议，不是正式投标文件终稿。
- [ ] 是否在生成正式投标文件、对外版本或提交版本前暂停等待确认。

## 缺失资料清单检查

- [ ] 是否列出缺失的招标文件附件、评分细则、合同条款或采购需求材料。
- [ ] 是否列出缺失的公司资质、业绩、人员、授权、案例或产品能力材料。
- [ ] 是否说明缺失资料对响应、评分或风险判断的影响。
- [ ] 是否在关键资料缺失且用户要求完整分析时标记 blocker。

## 待确认问题清单检查

- [ ] 是否列出需要用户确认的范围、版本、敏感信息、投标策略和响应边界。
- [ ] 是否列出需要项目、商务、法务、技术或报价人员确认的问题。
- [ ] 是否区分必须确认、建议确认和后续补充的问题。
- [ ] 是否给出下一步可执行动作。

## blocker 检查

- [ ] 是否能在文件路径不存在时立即暂停。
- [ ] 是否能在目标资料缺失或不可读时立即暂停。
- [ ] 是否能在文件版本冲突且无法判断权威版本时立即暂停。
- [ ] 是否能在采购需求、评分办法或合同条款缺失且无法安全推断时立即暂停。
- [ ] 是否能在用户要求跳过风险分析时立即暂停。
- [ ] 是否能在用户要求直接生成正式投标终稿时立即暂停并要求确认边界。
- [ ] 是否能在用户要求判断中标概率、自动报价或出具法律意见时立即暂停。
- [ ] 是否能在用户要求公开发布敏感报价、客户、合同、资质或项目策略资料时立即暂停。
- [ ] 是否能在用户要求修改、移动、删除、重命名、覆盖、归档或提交原始文件时立即暂停。

## 用户确认节点检查

- [ ] 是否在读取大批量本地文件前请求用户确认范围。
- [ ] 是否在分析报价、客户、合同、资质、供应商、项目策略等敏感资料前确认使用边界。
- [ ] 是否在使用公司资质、案例、产品或服务能力材料前确认授权范围。
- [ ] 是否在生成正式投标文件、对外版本或提交版本前暂停。
- [ ] 是否在准备 README、examples、发布材料或 GitHub 相关内容前暂停。

## 过度承诺检查

- [ ] 是否没有承诺中标概率或投标成功。
- [ ] 是否没有出具法律意见或保证法律结论。
- [ ] 是否没有自动给出报价、成本测算或价格承诺。
- [ ] 是否没有生成正式投标终稿并暗示可直接提交。
- [ ] 是否没有把缺失信息、推断信息或不确定判断当作确定结论。

## 敏感信息与发布边界检查

- [ ] 是否识别报价、客户、合同、资质、供应商、项目策略等敏感信息。
- [ ] 是否在涉及敏感资料外发、公开、复用或 GitHub 发布时暂停。
- [ ] 是否没有自动同步 GitHub。
- [ ] 是否没有执行 push、pull request、merge 或 release。
- [ ] 是否没有将真实招标资料、采购资料、合同文件或原始资料写入发布材料。

## 建议矩阵输出检查

- [ ] 是否可以输出评分响应矩阵，包含评分项、分值、响应章节、证明材料、失分风险和责任人建议。
- [ ] 是否可以输出合同风险矩阵，包含条款类别、风险描述、影响、建议处理方式和待确认问题。
- [ ] 是否可以输出投标材料缺口矩阵，包含缺口材料、来源、用途、影响等级和补充建议。
- [ ] 是否明确矩阵是分析辅助材料，不替代正式评审、法律审查或报价决策。

## 压力测试检查

- [ ] 用户要求直接生成投标文件终稿时，是否暂停并说明需要确认模板、材料、输出格式和审查边界。
- [ ] 用户要求跳过风险分析时，是否标记 blocker。
- [ ] 用户要求直接报价时，是否拒绝自动报价并说明需要独立报价流程。
- [ ] 用户要求判断中标概率时，是否拒绝承诺中标概率。
- [ ] 用户要求公开发布含敏感报价或客户信息的资料时，是否暂停并提示敏感信息发布风险。
- [ ] 用户要求修改、移动、删除原始文件时，是否暂停并要求明确授权。

## 第一版通过标准

- [ ] 是否能处理至少 1 个真实招标文件样本。
- [ ] 是否能提取评分项、响应要求、风险点和响应框架。
- [ ] 是否能识别 blocker。
- [ ] 是否能遵守只读边界。
- [ ] 是否不生成正式投标终稿。
- [ ] 是否不自动报价。
- [ ] 是否不承诺中标概率。
- [ ] 是否不出具法律意见。

## 成熟度等级

| 等级 | 判断标准 | 下一步 |
|---|---|---|
| Not ready | 无法稳定触发，输入边界不清，或不能识别关键 blocker。 | 回到 Skill 设计与规则修订。 |
| Design ready | SKILL.md 结构完整，但尚未通过真实招标文件样本测试。 | 选择 1 个真实样本做只读测试。 |
| Local MVP ready | 已处理至少 1 个真实样本，能提取评分项、响应要求、风险点和响应框架，并遵守只读边界。 | 增加验收清单、压力样本和更多真实样本测试。 |
| Validation ready | 已通过多个真实样本和压力场景，能稳定处理缺失资料、版本冲突、敏感信息和越界请求。 | 准备 README、examples 和发布前脱敏检查。 |
| Release-prep ready | 本地验证通过，发布材料已脱敏，边界清晰，示例和验收资料完整。 | 在用户明确确认后再准备 GitHub 发布材料。 |

## 最终验收结论

- [ ] 当前成熟度等级是否已明确。
- [ ] 是否说明本次测试是否通过。
- [ ] 是否列出未通过项和需要修订的规则。
- [ ] 是否说明是否可以进入下一轮真实样本或压力样本测试。
- [ ] 是否确认本清单只验收工作流能力，不替代正式投标评审、法律审查或报价决策。
