# Obsidian 资料组知识化处理流程

## 1. 规则定位

本规则用于资料目录、多个文件或资料包的多类型知识化入库。入口流程、确认节点和短报告格式以 `Obsidian-智能归档入口规则.md` 为准。

默认执行 **Obsidian 完整闭环低消耗模式**：成果链路完整，执行说明压缩，处理目标是生成可直接入档的正式知识成果。

## 2. 完整链路

```text
只读资料分析
-> 主资料类型、次资料类型、混合资料关系判断
-> 资料索引
-> 按主资料类型生成主题笔记
-> MOC
-> 按卡片类型生成或更新必要知识卡片
-> 双链与标签
-> 上层 MOC 挂接
-> 入档前自检
-> 用户授权写入确认
-> 正式入档
-> reviewed 状态
-> 边界检查
```

除非用户明确要求轻量处理，不得省略资料类型识别、MOC、主题笔记、知识卡片判断、双链、标签、入档前自检和挂接判断。

用户主动放入知识库、提供目录或要求知识化处理的资料组，默认视为用户认为有价值的知识材料。Codex 不再反复判断“是否值得入库”，而是判断资料组应如何高质量知识化入档：生成或更新资料索引、项目笔记、研究笔记、文献笔记、技术笔记、产品笔记、政策标准笔记、商务笔记、方法笔记、扩展笔记、MOC、知识卡片、相关卡片链接，或与已有内容合并。

合理分析和生成后的成果不视为阶段性草稿。只要来源清楚、结构完整、分析合理、无明显事实错误、无权限或冲突 blocker，且用户已授权写入正式 KnowledgeBase，即可直接入档。`_Codex_Output` 仅用于测试、预览、异常报告、过程检查或用户明确要求的临时输出。

资料组处理不能只生成知识卡片后结束。未判断资料类型、主题笔记类型、MOC 挂接、资料索引、相关卡片链接、重复合并和旧标题干扰的，视为未完成闭环。

## 3. 资料分析重点

资料组处理时，token 优先用于：

- 调用 `material-type-decision-matrix.md` 识别主资料类型、次资料类型、混合资料关系、主版本、历史版本、重复资料、外部案例和待确认资料；
- 判断同一项目多文件、同一主题多来源、同一资料不同版本、技术 + 产品 + 商务混合资料、政策 + 标准 + 解读资料、可研 + 附件 + 数据表等关系；
- 按主资料类型决定主题笔记类型，按次资料类型决定补充章节、风险检查和知识卡片类型；
- 判断正式入档目录和上层 MOC；
- 提炼高质量知识卡片；
- 判断是否需要合并到已有卡片、项目笔记或其他主题笔记；
- 判断是否存在旧标题、旧卡片、旧分类干扰检索。
- 形成核心判断、机制拆解、业务价值、适用边界和可复用表达。

不要把篇幅耗在过程日志和规则复述上。

## 4. 阶段要求

### 4.1 只读资料分析

- 只读取用户授权范围；
- 建立资料清单和分类；
- 输出主资料类型、次资料类型、建议主题笔记类型、建议知识卡片类型、MOC 新建/挂接和合并/版本更新判断；
- 标注版本关系、重复资料、不可读文件、待确认事项；
- 不移动、删除、重命名、覆盖或修改原始资料。

### 4.2 资料索引

- 作为资料组入口；
- 记录来源、分类、摘要、关键词、核心判断、业务启发、适用边界、可复用方向、待确认事项；
- 连接后续项目笔记、MOC 和知识卡片。

### 4.3 主题笔记

- 按 `topic-note-quality-standard.md` 选择项目笔记、研究笔记、文献笔记、技术笔记、产品笔记、政策标准笔记、商务笔记、方法笔记或扩展笔记；
- 承载资料主线，不得把所有资料组强行处理为项目笔记；
- 主类型决定主题笔记骨架，次类型决定补充章节和知识卡片类型；
- 说明背景、核心判断、关键内容、版本关系、业务链条、产品能力映射、工作路径、风险边界和可复用表达；
- 列出可拆分知识卡片；
- 给出正式入档目录和上层 MOC 挂接建议。

### 4.4 MOC

- 作为主题入口；
- 连接资料索引、项目笔记和知识卡片；
- 检查是否已有同主题 MOC，避免重复创建；
- 给出上层 MOC 挂接建议。

### 4.5 知识卡片

- 首轮默认 3-5 张；
- 按 `knowledge-card-quality-standard.md` 判断判断卡片、方法卡片、模型卡片、场景卡片、要求卡片、风险卡片、表达卡片、对比卡片或扩展卡片；
- 每张卡片只表达一个稳定、单一、可复用知识点；
- 必须包含机制拆解、业务价值、适用边界和可复用表达；
- 必须来源明确、可追踪、可独立引用；
- 不批量拆分低质量卡片。

## 5. 入档前自检与短报告

正式入档前按 `quality-checklist.md` 完成必要自检，确认成果已经合理分析和生成。自检不形成复杂审批流程，也不把合理成果降级为阶段性草稿。若用户已经明确授权写入正式 KnowledgeBase，可在自检通过后直接入档；若未授权写入，则先请求用户确认写入范围。

短报告格式：

```markdown
## 执行结果
- 模式：完整闭环低消耗
- 新增：
- 修改：
- 入档位置：
- 挂接位置：
- 原文件：
- blocker：
- 待确认：
```

未获得正式 KnowledgeBase 写入授权时，只询问：

```text
是否确认写入正式 KnowledgeBase？
```

## 6. 正式入档

用户授权后一次性完成：

- 写入 KnowledgeBase；
- 状态改为 `reviewed`；
- 挂接上层 MOC；
- 补充统一标签；
- 如本次使用了 `_Codex_Output` 预览或测试，按用户授权清理对应临时文件；
- 做边界检查；
- 输出短报告。

入档前必须检查目标目录、同名冲突、来源追踪、双链关系、标签和待确认事项。目标不明确或存在冲突时立即停止。

## 7. 边界检查

边界检查只保留必要结论：

- 原始资料是否未修改；
- 外接硬盘原文件是否未移动；
- 是否未写临时脚本；
- 是否未创建无必要临时文件；
- 是否仅写入本次授权成果；
- 正式文件是否为 `reviewed`；
- MOC、主题笔记、资料索引、知识卡片双链是否存在；
- 标签和上层 MOC 挂接是否完成；
- `_Codex_Output` 是否仅用于测试、预览、异常报告、过程检查或用户明确要求。

无异常不展开详细检查过程。

## 8. Blocker

出现 blocker 立即停止，按入口规则输出 Blocker 报告，并写入 `ERROR_LOG.md`。

## 9. 禁止事项

- 禁止修改原始资料；
- 禁止移动、删除、重命名外接硬盘原文件；
- 禁止写临时脚本；
- 禁止创建无必要临时文件；
- 禁止未授权写入正式库；
- 禁止绕过 blocker；
- 禁止重复创建同主题 MOC；
- 禁止把外部案例写成主项目事实；
- 禁止把候选版本写成正式版本；
- 禁止把用户主动要求处理的资料直接判定为“低价值不入库”。对重复、临时、来源不明、命名不清的资料，应判断是否合并、作为项目笔记来源、作为附件来源说明、建立关联、进入待确认区，或需要用户确认后再写入正式库。
