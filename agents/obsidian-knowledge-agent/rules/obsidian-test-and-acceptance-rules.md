# Obsidian 测试用例与验收规则

# 文件用途

本文件用于定义 Obsidian 规则闭环的测试输入、测试输出、静态检查、知识质量检查、自动修正和验收报告标准。

# 适用范围

适用于规则生成、规则补齐、测试样例生成、测试输出生成和闭环报告，不适用于正式资料入档。

# 输入要求

- 测试输入必须为虚构内容。
- 测试输入不得复制、改写或映射正式资料。
- 测试输入应覆盖项目背景、目标、任务、技术路线、交付成果、风险、待确认事项、来源说明、业务场景、产品能力和可复用表达线索。
- 多资料类型测试应覆盖项目资料、可研/研究报告、文献资料、技术文件、产品宣传/竞品资料、政策标准、商务资料、方法论/培训资料和混合资料。

# 输出要求

- 必须生成测试输入、测试输出和测试报告。
- 测试输出必须包含 YAML frontmatter、来源说明、状态说明、主体知识内容、双链、标签、未确认事项、下一步建议、自检结果和测试状态。
- 测试报告必须说明是否达到沙盒闭环可用状态，以及知识产出是否达到可入档、可复用、可支撑业务输出的质量要求。
- 测试报告必须引用 `quality-checklist.md` 完成资料类型识别、主题笔记质量、知识卡片质量、MOC 挂接、重复合并、证据来源和可复用性检查。

# frontmatter 要求

测试输出必须包含：

```yaml
review:
  human_checked: false
  formal_ingest_suggested: false
  formal_ingest_confirmed: false
test:
  is_test_output: true
  test_round: 1
```

# 正文结构要求

测试输出正文必须包含：

- 来源说明；
- 状态说明；
- 主体内容；
- 核心判断；
- 机制拆解或业务链条；
- 业务价值；
- 适用边界；
- 可复用表达；
- 关联知识；
- Obsidian 双链；
- 标签；
- 未确认事项；
- 下一步建议；
- 自检结果；
- 测试状态说明。

# 双链与标签要求

- 双链检查标准：是否存在 `[[页面名]]`。
- 标签检查标准：是否存在 `#测试` 和至少一个主题标签。

# 目录归属规则

测试目录结构固定为：

```text
<CODEX_OUTPUT_DIR>/_Test/input/
<CODEX_OUTPUT_DIR>/_Test/output/
<CODEX_OUTPUT_DIR>/_Test/report/
```

不得将测试文件写入正式知识库目录。`KnowledgeBase/_Codex_Output` 仅作为历史兼容说明，不再作为默认测试输出区。

# 禁止事项

- 禁止写临时脚本。
- 禁止绕过 blocker。
- 禁止修改 `Codex_Control`。
- 禁止修改原始资料。
- 禁止写入正式知识库内容。
- 禁止将测试输出标记为已人工检查、建议入档或确认入档。

# 自检清单

| 检查项 | 要求 |
|---|---|
| 结构检查 | frontmatter、标题、来源、状态、标签、双链齐全 |
| 合规检查 | 未写正式目录、未改原始资料、未改控制文件 |
| Obsidian 检查 | Markdown 可读、双链和标签格式正确 |
| 资料索引质量 | 有资料关系、核心判断、业务启发、适用边界和后续沉淀方向 |
| 项目笔记质量 | 有版本关系、业务链条、产品能力映射、方案表达和可拆分卡片 |
| 知识卡片质量 | 有机制拆解、业务价值、适用边界、可复用表达和关联知识 |
| 多资料类型覆盖 | 覆盖项目、研究、文献、技术、产品、政策标准、商务、方法和混合资料 |
| 主题笔记分型 | 验收项目笔记、研究笔记、文献笔记、技术笔记、产品笔记、政策标准笔记、商务笔记、方法笔记 |
| 知识卡片分型 | 验收判断卡片、方法卡片、模型卡片、场景卡片、要求卡片、风险卡片、表达卡片、对比卡片 |
| 统一质检 | 按 `quality-checklist.md` 输出通过、需修正、不适用或需人工确认 |
| 业务复用检查 | 能服务方案、可研、汇报、招采或客户沟通 |
| 反浅层检查 | 未停留在浅摘要、资料堆叠或概念摘录 |
| 修正检查 | 允许范围内问题已修正 |

## 多类型高质量验收要求

- 主题笔记只是摘要、资料堆叠或目录复述的，不能视为高质量入库成果。
- 知识卡片不能跨项目、跨场景复用，或缺少来源、适用场景、使用方式、边界、关联知识的，不能视为高质量知识卡片。
- 核心结论没有依据、没有证据、没有边界或没有风险标注的，不能通过知识质量验收。
- 政策标准测试必须区分“必须、应、宜、可”。
- 产品资料测试必须区分宣传话术、已验证能力、推断能力和待确认能力。
- 商务资料测试涉及法律性质结论时必须标注“需法务确认”。

## 内容深度与增强修正验收要求

入库闭环通过不等于知识质量通过。验收时必须单独判断主题笔记和知识卡片是否具备足够内容深度。

以下情况应判定为 P2：

- 入库状态、命名、frontmatter、标签、双链、MOC 和资料索引正确，但主题笔记只有浅摘要；
- 原始资料中的章节、表格、清单、技术路线、功能模块、实施计划、风险、指标或对比内容未被结构化利用；
- 主题笔记缺少问题、判断、路径、证据、复用价值或边界；
- 知识卡片缺少判断条件、方法框架、检查清单、使用方式、适用边界、反例或更新条件；
- 知识卡片只是定义、摘要、价值描述或宣传话术。

发现上述问题时，不得回退为 draft / migration 流程，也不得写入 `_Codex_Output`。应基于原始资料和已有正式成果执行正式成果增强修正，并在修正后重新验收。

# 验收标准

- 完成至少一轮静态检查。
- 完成至少一轮知识质量检查。
- 发现的问题已在允许范围内修正。
- 生成闭环测试报告。
- 报告必须分别说明流程闭环状态和知识质量状态。
- 报告结论只能为：已达到沙盒闭环可用状态且知识质量达标；部分达到，需要用户确认后继续完善；未达到，存在 blocker。
