# Obsidian 写入规则

## 1. 路径

Obsidian 知识库：

```text
<KNOWLEDGE_BASE_DIR>
```

临时输出区：

```text
<CODEX_OUTPUT_DIR>
```

说明：工作台 `_Codex_Output` 仅用于测试、预览、异常报告、过程检查、备份或用户明确要求的临时输出。`KnowledgeBase/_Codex_Output` 不作为默认写入目录；如历史存在，仅按用户明确确认进行清理或兼容处理。

过程记录目录：

```text
<RUN_LOG_DIR>
```

治理归档目录：

```text
<GOVERNANCE_ARCHIVE_DIR>
```

## 2. 默认模式

默认执行 **Obsidian formal knowledge filing 完整闭环低消耗模式**：

- 用户授权资料处理后，正式知识成果默认写入对应 KnowledgeBase 正式目录；
- `_Codex_Output` 仅用于测试、预览、异常报告、过程检查、备份或用户明确要求的临时输出；
- 保持资料类型识别、主题笔记类型判断、MOC、资料索引、知识卡片分型、双链、标签、frontmatter、上层 MOC 挂接和统一质检的完整成果标准；
- 项目笔记只是主题笔记的一类，不得把所有资料强行写成项目笔记；知识卡片必须是原子化深度知识卡片；
- 遇到 blocker、路径不明、权限问题、规则冲突、敏感信息风险、同名冲突或用户明确要求确认时，暂停并报告；
- 正式入档后一次性完成 `reviewed` 状态、挂接、统一标签、边界检查和短报告；
- 无异常只输出短报告。

## 3. 任务目标驱动与能力自补齐原则

写入 Obsidian 成果前，必须先判断用户最终需要的是知识库入库、方案报告、PPT、Word 文档、Excel 表、数据分析、图表、图片、代码、批量文件处理结果，还是项目资料整理成果。不得只按已有资料机械生成 Markdown。

当知识库和用户提供资料不足以支撑高质量成果时，应优先自行补齐：本地文件扫描、外部政策检索、行业资料检索、标准规范检索、政府官网资料检索、公开案例检索、公开数据检索、产品资料读取、数据清洗、图表 / 图片 / 示意图需求判断、占位说明、AI 图片生成提示词、代码处理和成品文件生成。只有业务口径、敏感案例、正式承诺、对外发布、产品最新状态无法判断时，才留给用户确认。

外部资料必须优先采用可信来源：国家部委官网、省级政府官网、住建厅 / 发改委 / 工信部门官网、地市州政府或主管部门官网、正式标准发布机构、权威行业协会、企业官网或正式产品资料、主流权威媒体报道。自媒体、营销号、来源不明转载内容只能作为线索，不得作为正式依据。

当成果涉及多阶段实施路径、多主体协同、平台架构、成果包组成、政策时间线、数据对比、项目闭环、系统流程、能力矩阵或推广路线时，应主动判断是否需要总体架构图、推广路径图、三方协同机制图、数字交付成果包结构图、试点项目最小闭环图、省平台—企业侧平台—项目侧关系图、政策依据时间线或能力矩阵表。Codex 只生成文字内容、结构排版、图片 / 图形占位符、图片说明和 AI 图片生成提示词，不执行图片生成、AI 作图、视觉渲染、封面图生成或架构图绘制。图片实际生成由 ChatGPT Image、image2、Midjourney、即梦、通义万相等其他图像智能体完成。Codex 不得因为无法生成图片而中断 Word、PPT、Markdown 或汇报材料输出。

正式写入前必须自检：内容是否深入、观点是否鲜明、政策依据是否充分、数据和案例是否可信、是否存在空泛表达、是否需要图表 / 图片 / 示意图占位但未说明、是否需要外部资料但未补充、是否把未核验内容写成确定事实、是否符合目标交付物格式、是否能直接用于后续工作。发现不足时，先自行补强再写入。

正式报告、方案、PPT 或知识库正文中，不应把“需补充资料”“需进一步核验事项”作为正文缺陷或结尾。资料缺口应处理为内部资料检索任务、附录、资料依据清单、内部待确认事项或后续工作台账。

正式任务完成后，应参考 [[任务目标驱动模式验收清单]] 进行结果自检。

## 4. 写入原则

- 新建 Obsidian 成果必须为 `.md`。
- 用户授权正式知识化入档且目标目录明确时，正式知识成果可写入对应 KnowledgeBase 正式目录。
- 测试、预览、异常报告、过程检查、备份或用户明确要求的临时输出，才写入 `<CODEX_OUTPUT_DIR>`。
- 不得覆盖已有 Markdown 文件。
- 不得删除、重命名、覆盖正式知识库文件。
- 不得修改原始资料。
- 不得写落盘临时脚本；允许使用已有工具、一次性命令或内联解释器命令完成扫描、统计、转换和校验，前提是不留下临时脚本文件。
- 如目标文件已存在，必须停止并报告 blocker，不得自动改名规避冲突。

KnowledgeBase 只保留用户授权的正式知识成果。验证产物、过程台账、临时报告、内部待确认事项、图表清单、外部资料准备清单、运行日志和中间输出不得写入 KnowledgeBase；其中属于测试、预览、异常报告、过程检查、备份或用户明确要求的临时输出，才可写入 `<CODEX_OUTPUT_DIR>`；运行日志和过程审计写入 `<RUN_LOG_DIR>`。不得混用 `_Codex_Output`、`AI输出成果` 和 `KnowledgeBase/_Codex_Output`。

KnowledgeBase 允许写入的范围仅限用户授权入档的正式 MOC、资料索引、主题笔记、项目笔记、研究笔记、文献笔记、技术笔记、产品笔记、政策标准笔记、商务笔记、方法笔记、未来扩展主题笔记、知识卡片，以及必要的关联、frontmatter、标签和双链更新。KnowledgeBase 禁止写入治理备份、过程台账、可用性验证输出、临时报告、草稿、外部资料检索清单、图表清单、内部待确认事项、运行日志、中间文件和未经用户确认的业务输出。

测试、预览、异常报告、过程检查、备份或用户明确要求的临时输出，才写入 `<CODEX_OUTPUT_DIR>`；正式知识成果不得默认写入 `_Codex_Output`。运行日志、治理记录、错误记录、过程审计默认输出到 `<RUN_LOG_DIR>`；治理备份默认输出到 `<GOVERNANCE_ARCHIVE_DIR>`。

## 5. 文件命名

```text
主题-用途-YYYYMMDD.md
```

示例：

- `示例平台建设项目-项目笔记-YYYYMMDD.md`
- `示例招采规则-知识卡片-YYYYMMDD.md`
- `示例平台-MOC-YYYYMMDD.md`

## 6. Markdown 基础模板

```yaml
---
title:
type:
status: reviewed
source:
source_path:
created:
updated:
tags:
  -
links:
  -
review:
  content_checked:
  source_traceable:
  formal_ingested:
---
```

正文最少包含知识内容，不包含处理过程：

```markdown
# 标题

## 来源说明
## 核心事实
## 结构化内容
## 关联笔记
## 分析备注
```

入档建议、人工检查状态、原文件是否移动、等待用户确认、处理状态、临时输出状态、执行日志等过程信息不得写入 MOC、主题笔记、项目笔记、资料索引或知识卡片正文。此类信息只能放入 frontmatter、单独 `_process` 元数据区或最终短报告。

来源说明必须保留反向打开原文件的能力：本地来源可保留可点击原文件链接，外部来源至少保留来源名称和路径/URL。

## 7. 正式入档规则

- 正式入档前必须输出建议入档目录和建议挂接的上层 MOC。
- 未获得正式 KnowledgeBase 写入授权时，不得写入正式知识库。
- 用户授权正式入档后，只写入本次生成或更新并通过检查的 Markdown 成果和必要关联更新。
- 正式入档后必须将正式文件状态设为 `reviewed`。
- 如本次因测试、预览、异常报告、过程检查、备份或用户明确要求而使用 `_Codex_Output`，后续清理必须另有用户授权，不得自动清理正式或临时文件。
- 正式入档必须满足：用户明确授权；明确归属正式知识类型；放入正确正式目录；不含过程记录、验证痕迹、内部待确认事项；完成 MOC 挂接或资料索引挂接；不污染正式知识库结构。

## 8. 短报告

正式入档或临时输出后使用：

```markdown
## 执行结果
- 模式：formal knowledge filing / 临时输出
- 新增：
- 修改：
- 建议入档：
- 建议挂接：
- 原文件：
- blocker：
- 待确认：
```

正式入库后使用：

```markdown
## 入库完成
- 入档：
- 状态：
- 挂接：
- 标签：
- 原文件：
- blocker：
```
