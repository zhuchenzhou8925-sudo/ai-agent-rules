# Obsidian 写入规则

## 1. 路径

Obsidian 知识库：

```text
<WORKSPACE_ROOT>/KnowledgeBase
```

默认草稿区：

```text
<CODEX_OUTPUT_DIR>
```

过程记录目录：

```text
<CODEX_CONTROL_DIR>/<RUN_LOG_DIR>
```

治理归档目录：

```text
<CODEX_CONTROL_DIR>/<GOVERNANCE_ARCHIVE_DIR>
```

## 2. 默认模式

默认执行 **Obsidian 完整闭环低消耗模式**：

- 草稿先写入 `<CODEX_OUTPUT_DIR>`；
- 保持 MOC、项目笔记或资料笔记、知识卡片、双链、标签建议、迁移建议、上层 MOC 挂接建议的完整成果标准；
- 项目笔记必须是全面资料笔记，知识卡片必须是原子化深度知识卡片；
- 用户确认前不迁入正式库；
- 用户确认后一次性完成迁移、`reviewed` 状态、挂接、统一标签、草稿清理和边界检查；
- 无异常只输出短报告。

## 3. 任务目标驱动与能力自补齐原则

写入 Obsidian 成果前，必须先判断用户最终需要的是知识库入库、方案报告、PPT、Word 文档、Excel 表、数据分析、图表、图片、代码、批量文件处理结果，还是项目资料整理成果。不得只按已有资料机械生成 Markdown。

当知识库和用户提供资料不足以支撑高质量成果时，应优先自行补齐：本地文件扫描、外部政策检索、行业资料检索、标准规范检索、政府官网资料检索、公开案例检索、公开数据检索、产品资料读取、数据清洗、图表 / 图片 / 示意图需求判断、占位说明、AI 图片生成提示词、代码处理和成品文件生成。只有业务口径、敏感案例、正式承诺、对外发布、产品最新状态无法判断时，才留给用户确认。

外部资料必须优先采用可信来源：国家部委官网、省级政府官网、住建厅 / 发改委 / 工信部门官网、地市州政府或主管部门官网、正式标准发布机构、权威行业协会、企业官网或正式产品资料、主流权威媒体报道。自媒体、营销号、来源不明转载内容只能作为线索，不得作为正式依据。

当成果涉及多阶段实施路径、多主体协同、平台架构、成果包组成、政策时间线、数据对比、项目闭环、系统流程、能力矩阵或推广路线时，应主动判断是否需要总体架构图、推广路径图、三方协同机制图、数字交付成果包结构图、试点项目最小闭环图、省平台—企业侧平台—项目侧关系图、政策依据时间线或能力矩阵表。Codex 只生成文字内容、结构排版、图片 / 图形占位符、图片说明和 AI 图片生成提示词，不执行图片生成、AI 作图、视觉渲染、封面图生成或架构图绘制。图片实际生成由 ChatGPT Image、image2、Midjourney、即梦、通义万相等其他图像智能体完成。Codex 不得因为无法生成图片而中断 Word、PPT、Markdown 或汇报材料输出。

正式写入前必须自检：内容是否深入、观点是否鲜明、政策依据是否充分、数据和案例是否可信、是否存在空泛表达、是否需要图表 / 图片 / 示意图占位但未说明、是否需要外部资料但未补充、是否把未核验内容写成确定事实、是否符合目标交付物格式、是否能直接用于后续工作。发现不足时，先自行补强再写入。

正式报告、方案、PPT 或知识库正文中，不应把“需补充资料”“需进一步核验事项”作为正文缺陷或结尾。资料缺口应处理为内部资料检索任务、附录、资料依据清单、内部待确认事项或后续工作台账。

正式任务完成后，应参考 [[任务目标驱动模式验收清单]] 进行结果自检。

## 4. 写入原则

- 新建 Obsidian 成果必须为 `.md`。
- 默认只向 `<CODEX_OUTPUT_DIR>` 写入新 Markdown 草稿。
- `<KNOWLEDGE_BASE_DIR>/_Codex_Output` 仅作为历史兼容区，不作为默认输出区。
- 不得覆盖已有 Markdown 文件。
- 不得删除、重命名、覆盖正式知识库文件。
- 不得修改原始资料。
- 不得写临时脚本。
- 自动化处理优先使用已有工具、一次性命令或内联命令；只有重复、稳定且经用户授权保留的场景才创建固定脚本。
- 如目标文件已存在，必须停止并报告 blocker，不得自动改名规避冲突。

KnowledgeBase 只保留用户确认后的正式知识成果。验证产物、过程台账、临时报告、内部待确认事项、图表清单、外部资料准备清单、运行日志和中间输出不得写入 KnowledgeBase；默认写入 `<CODEX_OUTPUT_DIR>` 或 `<CODEX_CONTROL_DIR>/<RUN_LOG_DIR>`。

KnowledgeBase 允许写入的范围仅限用户确认迁入的正式 MOC、项目笔记、资料笔记、知识卡片、正式资料索引，以及用户确认作为正式知识成果保留的文件。KnowledgeBase 禁止写入治理备份、过程台账、可用性验证输出、临时报告、草稿、外部资料检索清单、图表清单、内部待确认事项、运行日志、中间文件和未经用户确认的业务输出。

验证任务、报告增强任务、资料补齐任务、外部检索任务、图形化表达需求判断任务默认输出到 `<CODEX_OUTPUT_DIR>`；运行日志、治理记录、错误记录、过程审计默认输出到 `<CODEX_CONTROL_DIR>/<RUN_LOG_DIR>`；治理备份默认输出到 `<CODEX_CONTROL_DIR>/<GOVERNANCE_ARCHIVE_DIR>`。

## 5. 文件命名

```text
主题-用途-YYYYMMDD.md
```

示例：

- `湖北SaaS平台-项目笔记-20260616.md`
- `招采规则-知识卡片-20260616.md`
- `中信数智CDI平台-MOC-20260617.md`

## 6. Markdown 基础模板

```yaml
---
title:
type:
status: draft
source:
source_path:
created:
updated:
tags:
  -
links:
  -
review:
  human_checked: false
  migration_suggested: false
  migration_confirmed: false
---
```

正文最少包含知识内容，不包含处理过程：

```markdown
# 标题

## 来源说明
## 核心事实
## 结构化内容
## 关联笔记
## 分析备注
```

迁移建议、人工检查状态、原文件是否移动、等待用户确认、处理状态、草稿状态、执行日志等过程信息不得写入 MOC、项目笔记、知识卡片正文。此类信息只能放入 frontmatter、单独 `_process` 元数据区或最终短报告。

来源说明必须保留反向打开原文件的能力：本地来源可保留可点击原文件链接，外部来源至少保留来源名称和路径/URL。

## 7. 迁移规则

- 迁移前必须输出建议迁入目录和建议挂接的上层 MOC。
- 用户确认前不得迁入正式知识库。
- 用户确认后只迁移本次生成并通过检查的 Markdown 成果。
- 迁移后必须将正式文件状态改为 `reviewed`，并清理 `<CODEX_OUTPUT_DIR>` 对应草稿。
- 正式迁入必须满足：用户明确确认保留；明确归属正式知识类型；放入正确正式目录；不含过程记录、验证痕迹、内部待确认事项；完成 MOC 挂接或资料索引挂接；不污染正式知识库结构。
