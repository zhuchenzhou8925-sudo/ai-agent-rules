# Obsidian 知识库总规则

## 1. 规则定位

本文件统一 Obsidian 知识库成果的写作、正式入档、标签、双链和安全边界。入口流程以 `Obsidian-智能归档入口规则.md` 为准，不再以临时输出中转为默认流程。

默认模式为：**Obsidian 完整闭环低消耗模式**。该模式保留完整闭环成果，只压缩执行过程、规则读取和报告长度。

## 2. 默认读取范围

每次 Obsidian 入库任务默认只读取：

1. `Codex_Control/CODEX.md`
2. `Codex_Control/ERROR_LOG.md`
3. 当前 Obsidian 入库入口规则
4. 与任务直接相关的专项规则

不得每次无差别读取全部 Obsidian 规则。只有涉及正式入档、目录落位或同名冲突判断时，才读取 `directory-map.md` 并只读检查目标目录。

## 3. 成果标准

完整闭环任务必须生成或提供：

- 资料索引；
- MOC；
- 按资料类型生成的主题笔记或资料笔记；
- 知识卡片；
- 双链；
- frontmatter links；
- 标签建议；
- 正式入档目录建议；
- 上层 MOC 挂接建议；
- 输出复核报告；
- 用户授权后的正式入档、`reviewed` 状态、上层 MOC 挂接、统一标签和边界检查。

除非用户明确要求轻量处理，不得降级为只做摘要、只做索引或只做清单。

主题笔记必须按资料类型使用对应质量标准，不得把所有资料强行处理为项目笔记。项目笔记必须服务项目型资料，研究、文献、技术、产品、政策标准、商务、方法等资料应按对应主题笔记模板处理。知识卡片必须是原子化深度知识卡片，每张卡片只讲一个明确知识点，并讲清核心结论、类型、依据、场景、使用方式、边界、关联知识和更新条件。

## 4. 任务目标驱动与能力自补齐原则

Obsidian 知识库任务不得只基于已有知识库内容机械输出，必须先判断用户最终需要的是知识库入库、方案报告、PPT、Word 文档、Excel 表、数据分析、图表、图片、视频脚本、代码、批量文件处理结果，还是项目资料整理成果。

当知识库、原始资料或用户提供信息不足以支撑高质量成果时，不得简单写“资料不足”“建议补充资料”“需人工提供”。应优先自行补齐：外部政策检索、行业资料检索、标准规范检索、政府官网资料检索、公开案例检索、公开数据检索、产品资料读取、本地文件扫描、数据清洗、图表 / 图片 / 示意图需求判断、占位说明、AI 图片生成提示词、代码处理和成品文件生成。

外部资料应优先采用国家部委官网、省级政府官网、住建厅 / 发改委 / 工信部门官网、地市州政府或主管部门官网、正式标准发布机构、权威行业协会、企业官网或正式产品资料、主流权威媒体报道。自媒体、营销号、来源不明转载内容只能作为线索，不得作为正式依据。

当成果涉及多阶段实施路径、多主体协同关系、平台架构关系、成果包组成、政策时间线、数据对比、项目闭环、系统流程、能力矩阵或推广路线时，应主动判断是否需要图表、图片或示意图。Codex 只生成文字内容、结构排版、图片 / 图形占位符、图片说明和 AI 图片生成提示词，不执行图片生成、AI 作图、视觉渲染、封面图生成或架构图绘制。图片实际生成由 ChatGPT Image、image2、Midjourney、即梦、通义万相等其他图像智能体完成。Codex 不得因为无法生成图片而中断 Word、PPT、Markdown 或汇报材料输出。

当任务涉及批量处理、数据统计、格式转换、图形占位说明、文件扫描、表格清洗或重复操作时，应优先使用已有工具、一次性命令或内联命令完成。只有在任务重复、规则稳定、输入输出明确且用户授权保留时，才创建长期固定脚本；禁止落盘散乱临时脚本。所有自动化处理仍必须遵守写入范围、安全边界和关键操作确认规则。

正式输出前必须自检：内容是否足够深入、观点是否鲜明、政策依据是否充分、数据和案例是否可信、是否存在空泛表达、是否需要图表 / 图片 / 示意图占位但未说明、是否需要外部资料但未补充、是否存在未核验却写成确定事实的内容、是否符合目标交付物格式、是否能直接用于用户后续工作。发现不足时，应自行补强后再输出。

正式报告、方案、PPT 或知识库正文中，不应把“需补充资料”“需进一步核验事项”作为正文结尾。资料缺口应处理为内部资料检索任务、附录、资料依据清单、内部待确认事项或后续工作台账。

以下事项必须交由用户确认：是否引用敏感或未公开案例、是否采用正式业务口径、是否承诺具体指标、是否确认产品模块最新状态、是否确认商务合作模式、是否对外发送 / 发布 / 提交、是否采用某个正式版本作为最终依据。除此之外，应尽量自行完成资料补齐、研究、图形化表达需求判断、代码处理和成果生成。

正式任务完成后，应参考 [[任务目标驱动模式验收清单]] 进行结果自检。

## 5. 写入范围

- 用户授权、目录明确且无 blocker 时，正式知识成果默认写入 KnowledgeBase。
- `_Codex_Output` 仅用于测试、预览、异常报告、过程检查、备份或用户明确要求的临时输出。
- 测试输出只能写入 `<CODEX_OUTPUT_DIR>/_Test/`。
- 规则文件仅在用户明确要求修订规则时可修改。

KnowledgeBase 只保留用户授权后的正式知识成果。验证产物、过程台账、临时报告、内部待确认事项、图表清单、外部资料准备清单、运行日志和中间输出不得写入 KnowledgeBase；如需保留，应写入 `<CODEX_OUTPUT_DIR>` 或 `<RUN_LOG_DIR>`。

KnowledgeBase 允许写入的范围仅限用户授权正式入档的正式 MOC、资料索引、主题笔记、项目笔记、研究笔记、文献笔记、技术笔记、产品笔记、政策标准笔记、商务笔记、方法笔记、未来扩展主题笔记、知识卡片，以及用户确认作为正式知识成果保留的文件。KnowledgeBase 禁止写入治理备份、过程台账、可用性验证输出、临时报告、测试文件、外部资料检索清单、图表清单、内部待确认事项、运行日志、中间文件和未经用户确认的业务输出。

测试、预览、异常报告、过程检查、备份或用户明确要求的临时输出，可写入 `<CODEX_OUTPUT_DIR>`；运行日志、治理记录、错误记录、过程审计默认输出到 `<RUN_LOG_DIR>`；治理备份默认输出到 `<GOVERNANCE_ARCHIVE_DIR>`。

任何输出成果如需进入 KnowledgeBase，必须同时满足：用户已授权正式知识化处理；明确归属正式知识类型；放入正确正式目录；不含过程记录、验证痕迹、内部待确认事项；完成 MOC 挂接或资料索引挂接；不污染正式知识库结构。

## 6. frontmatter 建议

```yaml
---
title:
type:
status: reviewed
source:
source_path:
created:
updated:
tags:
  -
links:
  -
review:
  content_checked:
  source_traceable:
  formal_ingested:
---
```

正式入档成果状态统一为：

```yaml
status: reviewed
review:
  content_checked: true
  source_traceable: true
  formal_ingested: true
```

测试、预览或临时输出必须保留临时输出标识，且不得标记为正式入档。

## 7. 正文要求

成果正文应包含：

- 来源说明；
- 核心事实；
- 结构化内容；
- 关联笔记；
- 必要的分析备注；
- 未确认但影响复用的知识边界。

主题笔记承载资料主线，MOC 承载导航和挂接，知识卡片承载单一可复用知识点，三者不得互相替代。

正文不得写入 Codex 执行过程信息，包括人工检查状态、入档建议、入档目录、是否等待用户确认、原文件是否移动、处理状态、临时输出状态、执行日志。此类信息如必须保留，只能放入 frontmatter、单独 `_process` 元数据区或最终短报告。

来源章节必须保留反向打开原文件的能力：至少包含原始文件名、原始路径或来源说明；本地文件可保留可点击链接。

分析判断可以保留，但必须放在独立“分析备注”区，不得混入知识定义、事实描述和案例描述。

## 8. 双链与标签

- 双链使用 `[[页面名]]`，不写 `.md` 后缀。
- 标签使用中文业务标签，至少包含内容类型、主题领域和状态标签。
- 正式入档后补充统一标签，避免同义标签散乱。
- 挂接上层 MOC 时，只挂接已经存在或本次确认入档的成果。

## 9. 正式入档规则

Obsidian / knowledge-management 任务不再默认执行临时输出中转的两阶段流程。

用户已授权资料知识化处理、目标目录明确且无 blocker 时，正式知识成果直接进入 KnowledgeBase，并一次性完成 `reviewed` 状态、MOC 挂接、资料索引、统一标签、双链和边界检查。

出现目录不明、同名冲突、权限问题、规则冲突、敏感信息风险、覆盖/删除/移动/重命名、外部发布或用户明确要求确认时，必须暂停并报告 blocker。`_Codex_Output` 不作为正式知识成果的默认中转区。

## 10. 禁止事项

- 禁止修改原始资料；
- 禁止移动、删除、重命名外接硬盘原文件；
- 禁止写临时脚本；
- 禁止创建无必要临时文件；
- 禁止绕过 blocker；
- 禁止未授权写入正式库；
- 禁止覆盖、删除、重命名正式知识库既有文件；
- 禁止重复创建同主题 MOC；
- 禁止把待确认事实写成确定结论；
- 禁止降低 MOC、主题笔记、项目笔记、知识卡片、标签、挂接和 `reviewed` 状态要求。

## 11. 默认报告

无异常时只输出入口规则规定的短报告。有 blocker 时输出 Blocker 报告并写入 `ERROR_LOG.md`。
