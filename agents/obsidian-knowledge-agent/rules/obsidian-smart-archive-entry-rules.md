# Obsidian 智能归档入口规则

## 1. 规则定位

本规则是 Obsidian 知识库入库任务的统一入口，负责识别资料类型、选择处理模式、调度专项规则、控制确认节点和安全边界。

默认模式为：**Obsidian formal knowledge filing 完整闭环低消耗模式**。

该模式不是降低成果标准，而是压缩执行过程：保持完整闭环成果，减少非必要规则读取、过程日志、重复确认和长篇报告，把 token 优先用于资料类型识别、正式入档判断、关联挂接和知识卡片质量。

## 2. 默认完整闭环目标

执行知识库入库任务时，默认完整闭环目标包括：

1. 只读原始资料；
2. 调用 `material-type-decision-matrix.md` 识别主资料类型、次资料类型、混合资料关系和版本关系；
3. 调用 `topic-note-quality-standard.md` 判断主题笔记类型，不把所有资料强行处理为项目笔记；
4. 调用 `knowledge-card-quality-standard.md` 判断是否需要生成或更新知识卡片及卡片类型；
5. 判断是否需要生成或更新 MOC；
6. 判断是否需要生成或更新项目笔记、研究笔记、文献笔记、技术笔记、产品笔记、政策标准笔记、商务笔记、方法笔记、扩展笔记或资料笔记；
7. 建立双链；
8. 提供标签建议；
9. 判断是否需要更新资料索引；
10. 判断是否需要建立相关卡片链接；
11. 判断是否与已有知识重复、是否需要合并到已有卡片或已有笔记；
12. 判断是否已有同主题主题笔记、项目笔记、资料笔记或相关 MOC；
13. 判断是否需要新增 MOC 或挂接已有 MOC；
14. 判断是否需要更新 frontmatter links；
15. 判断是否存在旧标题、旧卡片或旧分类干扰检索；
16. 调用 `quality-checklist.md` 执行统一质检；
17. 判断正式 KnowledgeBase 入档目录；
18. 提供上层 MOC 挂接建议；
19. 用户已授权、路径明确且无 blocker 时，正式知识成果进入 KnowledgeBase；
20. 正式文件状态改为 `reviewed`；
21. 挂接上层 MOC；
22. 补充统一标签；
23. 完成边界检查和输出复核报告。

除非用户明确要求“只分析 / 只做摘要 / 只生成索引 / 轻量处理”，不得把完整闭环降级为摘要或轻量笔记。

只生成知识卡片但未判断主题笔记类型、资料索引、MOC 挂接、资料来源、合并去重和相关卡片链接的，视为未完成入库闭环。

## 3. 任务目标驱动与能力自补齐原则

执行 Obsidian 知识库任务时，必须以用户最终交付目标为核心，先判断本次任务最终需要的是知识库入库、方案报告、PPT、Word 文档、Excel 表、数据分析、图表、图片、视频脚本、代码、批量文件处理结果，还是项目资料整理成果。不得只根据用户已有资料或知识库已有内容机械输出。

### 3.1 知识库不足时主动补齐

当知识库内容、原始资料或用户提供信息不足以支撑高质量成果时，不得简单写“资料不足”“建议补充资料”“需人工提供”。应优先自行尝试补齐：

- 外部政策检索；
- 行业资料检索；
- 标准规范检索；
- 政府官网资料检索；
- 公开案例检索；
- 公开数据检索；
- 产品资料读取；
- 本地文件扫描；
- 数据清洗；
- 图表、图片和示意图的需求判断、占位说明和提示词设计；
- 代码处理；
- Markdown、Word、PPT、Excel、PDF 等成品文件生成。

只有涉及业务口径、敏感或未公开案例、正式承诺、商务合作模式、对外发布或提交、产品模块最新状态、正式版本依据无法判断时，才留给用户确认。

### 3.2 外部资料来源优先级

外部检索应优先使用可信来源：

1. 国家部委官网；
2. 省级政府官网；
3. 住建厅、发改委、工信部门官网；
4. 地市州政府或主管部门官网；
5. 正式标准发布机构；
6. 权威行业协会；
7. 企业官网或正式产品资料；
8. 主流权威媒体报道；
9. 其他来源仅作为线索，不作为正式依据。

不得优先使用自媒体、营销号、来源不明转载内容。外部资料进入正式成果前，应先整理为资料笔记、引用依据清单或附录依据，不得粗暴堆入正文。

### 3.3 Word / PPT / Markdown 文档中的图片生成边界规则

在基于知识库输出 Word、PPT、Markdown、汇报材料、方案文档等成果时，Codex 不执行图片生成、AI 作图、视觉渲染、封面图生成、架构图绘制等图片生成类工作。

Codex 只负责完成以下工作：

1. 判断文档中是否需要图形、图片、架构图、流程图、封面图、场景图、信息图、图标组合、页面原型示意等非文字内容；
2. 在文档对应位置保留清晰的图片或图形占位符；
3. 说明该图片或图形的使用位置、表达目的、核心内容、版式结构、风格要求；
4. 输出可复制给其他 AI 图像工具使用的图片生成提示词；
5. 继续完成文字内容、结构排版和 Word、PPT、Markdown 或汇报材料输出。

图片实际生成由其他 AI 图像智能体完成，例如 ChatGPT Image、image2、Midjourney、即梦、通义万相等。

Codex 不得因为无法生成图片而中断 Word、PPT、Markdown 或汇报材料输出。

本规则覆盖但不限于：

- 基于知识库生成 Word 报告；
- 基于知识库生成 PPT 大纲；
- 基于知识库生成 PPT 页面内容；
- 基于知识卡片生成汇报材料；
- 基于项目资料生成方案文档；
- 生成架构图说明；
- 生成流程图说明；
- 生成封面图说明；
- 生成智能建造、BIM、AI Agent、数字孪生、智慧工地等领域的图形化表达需求。

如文档需要图片，应使用以下占位格式：

```markdown
【图片 / 图形占位符：名称】

使用位置：
说明该图片用于第几页、第几章或哪个内容模块。

表达目的：
说明该图片要解决什么表达问题。

核心内容：
列出画面中需要呈现的主要元素。

推荐版式：
说明采用横向架构图、纵向流程图、分层结构图、关系图、场景插画、封面视觉图等形式。

风格要求：
说明配色、风格、比例、是否适合 PPT、是否偏正式汇报等。

AI 图片生成提示词：
提供一段可直接复制给其他 AI 图像工具的完整提示词。
```

### 3.4 代码和自动化处理规则

当任务涉及批量处理、数据统计、格式转换、图形占位说明、文件扫描、表格清洗或重复操作时，应优先调用已有工具、一次性命令或内联解释器命令完成；确需长期复用时，才在用户确认后创建固定脚本。可自动化处理的事项包括扫描目录、生成文件清单、统计 Markdown / Word / PDF / Excel 文件、清洗 Excel / CSV、生成数据表、生成图形占位说明和 AI 图片生成提示词、批量生成 Markdown、批量检查链接、批量检查结构缺项、批量生成 Word / PPT / PDF、生成对照表和台账。

脚本创建和写入仍必须遵守 `CODEX.md` 的脚本规则、写入权限规则和关键操作确认规则。禁止落盘创建临时脚本或一次性脚本目录。

除非用户明确授权保留为长期工具，不得为单次任务写临时脚本。一次性处理应使用已有工具、一次性命令或内联命令完成，并且不留下散落过程脚本。

### 3.5 正式输出前自检

正式输出前必须自行检查：

- 内容是否足够深入；
- 观点是否鲜明；
- 政策依据是否充分；
- 数据和案例是否可信；
- 是否存在空泛表达；
- 是否需要图表、图片或示意图占位但未说明；
- 是否需要外部资料但未补充；
- 是否存在未核验却写成确定事实的内容；
- 是否符合目标交付物格式；
- 是否能直接用于用户后续工作。

发现不足时，应自行补强后再输出。

### 3.6 不得把资料缺口写成正文缺陷

正式报告、方案、PPT 或知识库正文中，不应把“需补充资料”“需进一步核验事项”作为正文结尾。资料缺口应处理为内部资料检索任务、附录、资料依据清单、内部待确认事项或后续工作台账。正式正文应保持完整，不应让读者感觉报告尚未准备好。

资料缺口、验证产物、过程台账、临时报告、内部待确认事项、图表清单、外部资料准备清单、运行日志和中间输出不得写入 KnowledgeBase。上述内容如需保留，应写入 `<CODEX_OUTPUT_DIR>` 或 `<RUN_LOG_DIR>`；治理备份默认写入 `<GOVERNANCE_ARCHIVE_DIR>`。

正式任务完成后，应参考 [[任务目标驱动模式验收清单]] 进行结果自检。

## 5. 输入与模式识别

支持用户直接输入：

```text
分析：/path/to/file-or-folder
整理：/path/to/file-or-folder
入库：/path/to/file-or-folder
归档：/path/to/file-or-folder
```

模式判断：

| 用户意图 | 执行模式 |
|---|---|
| 分析、看看、扫描、判断、只读 | 只读分析模式：只输出清单、分类、处理建议 |
| 整理、知识化、预览、测试、用户明确要求临时输出 | 临时输出模式：仅在测试、预览、异常报告、过程检查、备份或用户明确要求时写入 `<CODEX_OUTPUT_DIR>` |
| 入库、归档、自动闭环、完整处理、分析并归档 | Obsidian formal knowledge filing 完整闭环低消耗模式：用户授权、路径明确且无 blocker 时，正式知识成果直接进入 KnowledgeBase |

用户明确指定轻量范围时，按用户范围执行；否则入库和归档默认保持完整闭环目标。

## 6. 输入对象识别

处理前必须只读识别：

- 路径是否存在、可读取、在用户授权范围内；
- 输入是单文件、多个文件、资料目录还是已生成临时输出；
- 文件类型、大小、修改时间、来源路径；
- 资料主要属于项目资料、可研/研究报告、文献资料、技术文件、产品宣传/产品手册/竞品资料、政策标准/指南/规范、商务资料/报价/合同/招采文件、培训资料/方法论/操作手册、混合资料或未来扩展资料类型中的哪类；
- 资料是否存在次要资料类型，以及主类型与次类型如何影响主题笔记、补充章节和知识卡片类型；
- 是否存在主版本、历史版本、重复资料、外部案例、待确认资料；
- 是否已有同主题 MOC 或同主题知识卡片。
- 是否已有同主题主题笔记、项目笔记或资料笔记。
- 如用户未说明具体归档区域，Codex 必须自动判断建议归档区域，并说明判断理由。

资料入库判断原则：

用户主动放入知识库、提供路径或要求知识化处理的材料，默认视为用户认为有价值的知识材料。Codex 不再反复判断“是否值得入库”，而是自动判断“如何入库”。

每次知识化处理必须自动判断：

- 主资料类型和次资料类型；
- 是否属于混合资料、版本更新或同主题多来源；
- 建议生成的主题笔记类型；
- 是否需要生成或更新知识卡片；
- 建议知识卡片类型；
- 是否需要生成或更新项目笔记、研究笔记、文献笔记、技术笔记、产品笔记、政策标准笔记、商务笔记、方法笔记、扩展笔记或资料笔记；
- 是否需要挂接 MOC；
- 是否已有相关 MOC；
- 是否需要新增 MOC；
- 是否需要更新资料索引；
- 是否需要建立相关卡片链接；
- 是否需要双链挂接；
- 是否需要更新 frontmatter links；
- 是否与已有知识重复；
- 是否需要合并到已有卡片；
- 是否存在旧标题、旧卡片、旧分类干扰检索。

入库方式判断优先级：

1. 先按 `material-type-decision-matrix.md` 判断主资料类型和次资料类型；
2. 主资料类型决定主题笔记类型；
3. 次资料类型决定补充章节、风险检查和知识卡片类型；
4. 主题导航型资料组：生成或挂接 MOC；
5. 稳定单一方法、概念、判断标准、模型、要求、风险、表达或对比：按卡片分型生成或更新知识卡片；
6. 多文件资料组：先生成资料索引，再按主资料类型生成主题笔记、MOC 和必要的小批量知识卡片；
7. 重复、临时、来源不明、命名不清的材料：不得直接判定为“不入库”，应判断是否合并到已有卡片或笔记、只作为主题笔记来源、作为附件来源说明、建立关联、进入待确认区，或需要用户确认后再正式入档。

知识卡片应优先沉淀抽象概念、方法、模式、逻辑和可复用话术。具象主体、项目名称、比赛 title、单位名称、时间、来源背景、申报背景、实施细节，应优先放入项目笔记、案例笔记、来源说明或相关卡片中，不应主导通用知识卡片命名。

如暂不生成主题笔记、项目笔记、案例笔记、资料笔记或暂不挂接 MOC，必须在输出报告中说明原因。

只生成知识卡片不算完整闭环。完整入库闭环至少要判断资料类型、主题笔记类型、资料索引、主题笔记或资料笔记、知识卡片、MOC 挂接、相关链接、frontmatter、正式入档状态和输出复核报告。

## 7. 确认节点与暂停条件

Obsidian / knowledge-management 任务不再默认执行临时输出中转的两阶段流程。

用户已授权资料知识化处理、目标目录可判断且无 blocker 时，默认执行 formal knowledge filing：正式知识成果直接进入 KnowledgeBase，状态使用 `reviewed`，并同步完成 MOC、资料索引、主题笔记、知识卡片、frontmatter、标签和双链闭环。

只有出现以下情况时才暂停并报告：

- 用户明确要求只读分析、测试、预览、临时输出或先确认；
- 输入路径不存在、不可读或超出授权范围；
- 目标正式目录无法判断；
- 目标文件同名冲突或版本关系影响结论；
- 资料类型、主题笔记类型或合并去重判断无法成立；
- 需要覆盖、删除、移动、重命名原始资料或正式库既有文件；
- 需要新增未定义正式目录；
- 存在权限问题、规则冲突、敏感信息风险或对外发布风险；
- 需要 GitHub 同步、外部发布或其他高风险操作；
- 需要绕过本规则安全边界。

`_Codex_Output` 仅用于测试、预览、异常报告、过程检查、备份或用户明确要求的临时输出，不作为正式知识成果的默认中转区。

## 8. formal knowledge filing 生成要求

正式入档成果必须满足：

- 基于原始资料只读分析，不修改原始资料；
- 保留来源路径、来源说明和必要的可点击原文件链接；
- 正式成果使用 `status: reviewed`；
- 明确资料类别、主题笔记类型、核心判断、未确认事项和适用场景；
- MOC、主题笔记、资料索引、知识卡片之间建立双链；
- 知识卡片必须主题单一、来源明确、可独立复用；
- 标签至少包含内容类型、主题领域、状态标签；
- 给出正式入档目录判断和上层 MOC 挂接建议；
- 检查同主题 MOC、主题笔记、资料索引和知识卡片，避免重复创建。

正文质量必须满足：

- 主题笔记或资料笔记必须符合 `topic-note-quality-standard.md`，不得停留在几段概括性摘要；
- 项目笔记必须按项目型资料结构展开；研究、文献、技术、产品、政策标准、商务、方法等资料必须按对应主题笔记模板展开；
- 知识卡片必须以 `knowledge-card-quality-standard.md` 为唯一结构标准，每张卡片只讲一个明确知识点，并覆盖卡片类型、核心结论、来源依据、适用场景、使用方式、边界、关联知识和更新条件；
- MOC、主题笔记、知识卡片正文不得写入 Codex 执行过程信息；
- 入档建议、人工检查状态、处理状态、临时输出状态、原文件是否移动、等待用户确认、执行日志等只能进入 frontmatter、单独 `_process` 元数据区或最终短报告；
- 分析判断必须进入独立“分析备注”区，不得混入知识定义和事实描述；
- 来源章节必须保留反向打开原文件的能力。

知识卡片首轮默认小批量生成，通常 3-5 张；宁可少而准，不批量拆分低质量卡片。

## 9. 正式入档要求

正式入档在用户授权、目录明确且无 blocker 时执行。入档前必须只读检查：

- 目标目录属于现有 KnowledgeBase 正式分类；
- 目标文件不存在同名冲突；
- 入档清单仅包含本次生成、更新或合并且通过检查的 Markdown 成果；
- 过程报告、分析中间件、测试文件、验证产物、临时报告、图表清单、外部资料准备清单、内部待确认事项、运行日志和中间文件不进入正式库；
- 原始资料不移动、不删除、不重命名、不覆盖；
- 需要新建目录时，目录必须属于明确的正式分类；不明确则停止。

正式进入 KnowledgeBase 必须同时满足：用户已授权正式知识化处理；明确归属正式 MOC、资料索引、主题笔记、项目笔记、研究笔记、文献笔记、技术笔记、产品笔记、政策标准笔记、商务笔记、方法笔记、未来扩展主题笔记、知识卡片或其他正式知识成果；放入正确正式目录；不含过程记录、验证痕迹和内部待确认事项；完成 MOC 挂接或资料索引挂接；不污染正式知识库结构。

入档后必须：

- 将正式成果 `status` 保持为 `reviewed`；
- 更新 MOC 双链和上层 MOC 挂接；
- 补充统一标签；
- 确认原始资料未修改；
- 调用 `quality-checklist.md` 做边界检查。

如果测试、预览、异常报告、过程检查、备份或用户明确要求导致使用 `_Codex_Output`，只能按用户授权清理对应临时输出，不得把临时输出清理作为正式入档的默认步骤。

## 10. 短报告格式

正式入档或临时输出阶段默认只输出：

```markdown
## 执行结果
- 模式：formal knowledge filing / 临时输出
- 新增：
- 修改：
- 建议入档：
- 建议挂接：
- 原文件：
- blocker：
- 待确认：
```

正式入库阶段默认只输出：

```markdown
## 入库完成
- 入档：
- 状态：
- 挂接：
- 标签：
- 原文件：
- blocker：
```

无异常时不得展开：

- 完整规则解释；
- 全部读取文件清单；
- 每一步执行过程；
- 未修改文件长列表；
- 未创建临时文件的长篇说明；
- 详细链路检查过程。

## 11. Blocker 处理

发生 blocker 必须立即停止，不得绕过，并按既有规则写入 `ERROR_LOG.md`。

正式入档、覆盖、删除、重命名、归档、GitHub 操作、本机 Skill 修改等关键写入动作前，必须确认用户授权；授权不明或存在高风险时必须暂停等待用户确认。

输出格式：

```markdown
## Blocker
- 问题：
- 停止位置：
- 已完成：
- 未执行：
- 建议处理：
```

Blocker 包括但不限于：

- 输入路径不存在、不可读或超出授权范围；
- 需要修改原始资料；
- 需要移动、删除、重命名或覆盖原始资料；
- 目标正式目录无法判断；
- 目标文件同名冲突；
- 资料版本边界不清且影响结论；
- 需要扫描未授权目录；
- 需要写临时脚本或无必要临时文件；
- 命令失败且影响范围无法确认；
- 需要绕过本规则安全边界。

## 12. 外接硬盘资料边界

路径位于 `/Volumes/...` 等外接硬盘时：

- 原文件只读处理，不移动、不删除、不重命名、不覆盖、不修改；
- 只读取用户明确提供的文件或目录，不扩大扫描；
- 仅在测试、预览、异常报告、过程检查、备份或用户明确要求时，临时输出才写入 `<CODEX_OUTPUT_DIR>`；
- 不使用 `KnowledgeBase/_Codex_Output` 作为默认临时输出区；
- 正式入档时，只处理本次生成、更新或合并的 Markdown 知识成果，不搬运外接硬盘原文件；
- 来源路径保留外接硬盘原路径；
- 外接硬盘未挂载或路径不可确认时，标注“原始文件路径待确认”，不得伪造链接。

## 13. 质量优先级

执行时 token 优先用于：

1. 按原始资料结构吸收资料主体内容；
2. 判断正式入档路径和上层 MOC；
3. 提炼稳定、单一、可复用且有深度的知识卡片；
4. 建立准确双链和标签；
5. 标注不确定事项。

不得把执行重点转移到长篇过程说明、规则复述或重复验证。

## 14. 安全边界

必须保留：

- 不修改原始资料；
- 不移动外接硬盘原文件；
- 不写临时脚本；
- 不创建无必要临时文件；
- 不绕过 blocker；
- 不在未授权情况下写入正式库；
- 不重复创建同主题 MOC；
- 不降低 MOC、主题笔记、项目笔记、知识卡片、标签、挂接和 `reviewed` 状态要求。
