# Obsidian 智能归档入口规则

## 1. 规则定位

本规则是 Obsidian 知识库入库任务的统一入口，负责识别资料类型、选择处理模式、调度专项规则、控制确认节点和安全边界。

默认模式为：**Obsidian 完整闭环低消耗模式**。

该模式不是降低成果标准，而是压缩执行过程：保持完整闭环成果，减少非必要规则读取、过程日志、重复确认和长篇报告，把 上下文用量 优先用于资料深度吸收、归档判断和知识卡片质量。

## 2. 默认完整闭环目标

执行知识库入库任务时，默认完整闭环目标包括：

1. 只读原始资料；
2. 生成 MOC；
3. 生成项目笔记或资料笔记；
4. 生成知识卡片；
5. 建立双链；
6. 提供标签建议；
7. 提供正式库迁移建议；
8. 提供上层 MOC 挂接建议；
9. 用户确认后迁入 KnowledgeBase；
10. 正式文件状态改为 `reviewed`；
11. 挂接上层 MOC；
12. 补充统一标签；
13. 完成边界检查。

除非用户明确要求“只分析 / 只做摘要 / 只生成索引 / 轻量处理”，不得把完整闭环降级为摘要或轻量笔记。

## 3. 任务目标驱动与能力自补齐原则

执行 Obsidian 知识库任务时，必须以用户最终交付目标为核心，先判断本次任务最终需要的是知识库入库、方案报告、PPT、Word 文档、Excel 表、数据分析、图表、图片、视频脚本、代码、批量文件处理结果，还是项目资料整理成果。不得只根据用户已有资料或知识库已有内容机械输出。

### 3.1 知识库不足时主动补齐

当知识库内容、原始资料或用户提供信息不足以支撑高质量成果时，不得简单写“资料不足”“建议补充资料”“需人工提供”。应优先自行尝试补齐：

- 外部政策检索；
- 行业资料检索；
- 标准规范检索；
- 政府官网资料检索；
- 公开案例检索；
- 公开数据检索；
- 产品资料读取；
- 本地文件扫描；
- 数据清洗；
- 图表、图片和示意图的需求判断、占位说明和提示词设计；
- 代码处理；
- Markdown、Word、PPT、Excel、PDF 等成品文件生成。

只有涉及业务口径、敏感或未公开案例、正式承诺、商务合作模式、对外发布或提交、产品模块最新状态、正式版本依据无法判断时，才留给用户确认。

### 3.2 外部资料来源优先级

外部检索应优先使用可信来源：

1. 国家部委官网；
2. 省级政府官网；
3. 住建厅、发改委、工信部门官网；
4. 地市州政府或主管部门官网；
5. 正式标准发布机构；
6. 权威行业协会；
7. 企业官网或正式产品资料；
8. 主流权威媒体报道；
9. 其他来源仅作为线索，不作为正式依据。

不得优先使用自媒体、营销号、来源不明转载内容。外部资料进入正式成果前，应先整理为资料笔记、引用依据清单或附录依据，不得粗暴堆入正文。

### 3.3 Word / PPT / Markdown 文档中的图片生成边界规则

在基于知识库输出 Word、PPT、Markdown、汇报材料、方案文档等成果时，Codex 不执行图片生成、AI 作图、视觉渲染、封面图生成、架构图绘制等图片生成类工作。

Codex 只负责完成以下工作：

1. 判断文档中是否需要图形、图片、架构图、流程图、封面图、场景图、信息图、图标组合、页面原型示意等非文字内容；
2. 在文档对应位置保留清晰的图片或图形占位符；
3. 说明该图片或图形的使用位置、表达目的、核心内容、版式结构、风格要求；
4. 输出可复制给其他 AI 图像工具使用的图片生成提示词；
5. 继续完成文字内容、结构排版和 Word、PPT、Markdown 或汇报材料输出。

图片实际生成由其他 AI 图像智能体完成，例如 ChatGPT Image、image2、Midjourney、即梦、通义万相等。

Codex 不得因为无法生成图片而中断 Word、PPT、Markdown 或汇报材料输出。

本规则覆盖但不限于：

- 基于知识库生成 Word 报告；
- 基于知识库生成 PPT 大纲；
- 基于知识库生成 PPT 页面内容；
- 基于知识卡片生成汇报材料；
- 基于项目资料生成方案文档；
- 生成架构图说明；
- 生成流程图说明；
- 生成封面图说明；
- 生成智能建造、BIM、AI Agent、数字孪生、智慧工地等领域的图形化表达需求。

如文档需要图片，应使用以下占位格式：

```markdown
【图片 / 图形占位符：名称】

使用位置：
说明该图片用于第几页、第几章或哪个内容模块。

表达目的：
说明该图片要解决什么表达问题。

核心内容：
列出画面中需要呈现的主要元素。

推荐版式：
说明采用横向架构图、纵向流程图、分层结构图、关系图、场景插画、封面视觉图等形式。

风格要求：
说明配色、风格、比例、是否适合 PPT、是否偏正式汇报等。

AI 图片生成提示词：
提供一段可直接复制给其他 AI 图像工具的完整提示词。
```

### 3.4 代码和自动化处理规则

当任务涉及批量处理、数据统计、格式转换、图形占位说明、文件扫描、表格清洗或重复操作时，应优先使用已有工具、一次性命令或内联命令完成。只有在任务重复、规则稳定、输入输出明确且用户授权保留时，才创建长期固定脚本；禁止落盘散乱临时脚本。适用场景包括但不限于扫描目录、生成文件清单、统计 Markdown / Word / PDF / Excel 文件、清洗 Excel / CSV、生成数据表、生成图形占位说明和 AI 图片生成提示词、批量生成 Markdown、批量检查链接、批量检查结构缺项、批量生成 Word / PPT / PDF、生成对照表和台账。

脚本创建和写入仍必须遵守 `CODEX.md` 的脚本规则、写入权限规则和关键操作确认规则。

### 3.5 正式输出前自检

正式输出前必须自行检查：

- 内容是否足够深入；
- 观点是否鲜明；
- 政策依据是否充分；
- 数据和案例是否可信；
- 是否存在空泛表达；
- 是否需要图表、图片或示意图占位但未说明；
- 是否需要外部资料但未补充；
- 是否存在未核验却写成确定事实的内容；
- 是否符合目标交付物格式；
- 是否能直接用于用户后续工作。

发现不足时，应自行补强后再输出。

### 3.6 不得把资料缺口写成正文缺陷

正式报告、方案、PPT 或知识库正文中，不应把“需补充资料”“需进一步核验事项”作为正文结尾。资料缺口应处理为内部资料检索任务、附录、资料依据清单、内部待确认事项或后续工作台账。正式正文应保持完整，不应让读者感觉报告尚未准备好。

资料缺口、验证产物、过程台账、临时报告、内部待确认事项、图表清单、外部资料准备清单、运行日志和中间输出不得写入 KnowledgeBase。上述内容默认写入 `<CODEX_OUTPUT_DIR>` 或 `<CODEX_CONTROL_DIR>/<RUN_LOG_DIR>`；治理备份默认写入 `<CODEX_CONTROL_DIR>/<GOVERNANCE_ARCHIVE_DIR>`。`<KNOWLEDGE_BASE_DIR>/_Codex_Output` 仅作为历史兼容区，不作为默认输出区。

正式任务完成后，应参考 [[任务目标驱动模式验收清单]] 进行结果自检。

## 5. 输入与模式识别

支持用户直接输入：

```text
分析：/path/to/file-or-folder
整理：/path/to/file-or-folder
入库：/path/to/file-or-folder
归档：/path/to/file-or-folder
```

模式判断：

| 用户意图 | 执行模式 |
|---|---|
| 分析、看看、扫描、判断、只读 | 只读分析模式：只输出清单、分类、处理建议 |
| 整理、知识化、生成草稿、不迁移 | 草稿生成模式：生成完整草稿成果到 `<CODEX_OUTPUT_DIR>`，不迁入正式库 |
| 入库、归档、自动闭环、完整处理、分析并归档 | Obsidian 完整闭环低消耗模式：先生成完整草稿，用户确认后一次性迁入正式库 |

用户明确指定轻量范围时，按用户范围执行；否则入库和归档默认保持完整闭环目标。

## 6. 输入对象识别

处理前必须只读识别：

- 路径是否存在、可读取、在用户授权范围内；
- 输入是单文件、多个文件、资料目录还是已生成草稿；
- 文件类型、大小、修改时间、来源路径；
- 资料属于项目资料、产品能力、行业场景、商务资料、汇报材料、可复用知识中的哪类；
- 是否存在主版本、历史版本、重复资料、外部案例、待确认资料；
- 是否已有同主题 MOC 或同主题知识卡片。

资料价值判断优先级：

1. 项目型或事项型资料：生成项目笔记或资料笔记；
2. 主题导航型资料组：生成或挂接 MOC；
3. 稳定单一方法、概念、判断标准：生成知识卡片；
4. 多文件资料组：先生成资料索引，再生成项目笔记、MOC 和小批量知识卡片；
5. 低价值、重复、临时、来源不明资料：只给处理建议，不生成正式候选。

## 7. 两个确认节点

### 7.1 草稿生成后确认

Codex 先在 `<CODEX_OUTPUT_DIR>` 生成完整草稿成果，包括：

- MOC；
- 项目笔记或资料笔记；
- 知识卡片；
- 双链；
- 标签建议；
- 建议迁入目录；
- 建议挂接的上层 MOC。

此阶段不逐步确认。完成后只询问：

```text
是否确认迁入正式库？
```

### 7.2 正式迁移前确认

用户确认后，一次性完成：

- 迁入 KnowledgeBase；
- 状态改为 `reviewed`；
- 挂接上层 MOC；
- 补充统一标签；
- 清理 `<CODEX_OUTPUT_DIR>` 对应草稿；
- 做边界检查；
- 输出短报告。

除以上两个节点外，不在每个小步骤反复暂停确认。遇到 blocker 例外，必须立即停止。

## 8. 草稿生成要求

草稿默认写入：

```text
<CODEX_OUTPUT_DIR>
```

草稿必须满足：

- 基于原始资料只读分析，不修改原始资料；
- 保留来源路径、来源说明和必要的可点击原文件链接；
- 使用 `status: draft`；
- 明确资料类别、核心判断、未确认事项和适用场景；
- MOC、项目笔记、知识卡片之间建立双链；
- 知识卡片必须主题单一、来源明确、可独立复用；
- 标签至少包含内容类型、主题领域、状态标签；
- 给出正式迁移目录建议和上层 MOC 挂接建议；
- 检查同主题 MOC，避免重复创建。

正文质量必须满足：

- 项目笔记或资料笔记必须是全面资料笔记，不得停留在几段概括性摘要；
- 项目笔记必须按原始资料结构展开，尽量完整吸收关键事实、项目背景、技术路径、系统能力、场景描述、案例数据、产品能力和方法论；
- 知识卡片必须是原子化深度知识卡片，每张卡片只讲一个明确知识点，并覆盖定义、背景、机制、组成、路径、场景、边界、表达和来源；
- MOC、项目笔记、知识卡片正文不得写入 Codex 执行过程信息；
- 迁移建议、人工检查状态、处理状态、草稿状态、原文件是否移动、等待用户确认、执行日志等只能进入 frontmatter、单独 `_process` 元数据区或最终短报告；
- 分析判断必须进入独立“分析备注”区，不得混入知识定义和事实描述；
- 来源章节必须保留反向打开原文件的能力。

知识卡片首轮默认小批量生成，通常 3-5 张；宁可少而准，不批量拆分低质量卡片。

## 9. 正式迁移要求

正式迁移只在用户确认后执行。迁移前必须只读检查：

- 目标目录属于现有 KnowledgeBase 正式分类；
- 目标文件不存在同名冲突；
- 迁移清单仅包含本次生成且通过检查的 Markdown 成果；
- 过程报告、分析中间件、测试文件、验证产物、临时报告、图表清单、外部资料准备清单、内部待确认事项、运行日志和中间文件不迁移；
- 原始资料不移动、不删除、不重命名、不覆盖；
- 需要新建目录时，目录必须属于明确的正式分类；不明确则停止。

正式迁入 KnowledgeBase 必须同时满足：用户明确确认保留；明确归属正式 MOC、项目笔记、资料笔记、知识卡片、正式资料索引或其他正式知识成果；放入正确正式目录；不含过程记录、验证痕迹和内部待确认事项；完成 MOC 挂接或资料索引挂接；不污染正式知识库结构。

迁移后必须：

- 将正式成果 `status` 改为 `reviewed`；
- 删除或清理 `<CODEX_OUTPUT_DIR>` 中对应草稿；
- 更新 MOC 双链和上层 MOC 挂接；
- 补充统一标签；
- 确认原始资料未修改；
- 做边界检查。

## 10. 短报告格式

草稿生成阶段默认只输出：

```markdown
## 执行结果
- 模式：完整闭环低消耗
- 新增：
- 修改：
- 建议迁入：
- 建议挂接：
- 原文件：
- blocker：
- 待确认：
```

正式入库阶段默认只输出：

```markdown
## 入库完成
- 迁移：
- 状态：
- 挂接：
- 标签：
- 原文件：
- blocker：
```

无异常时不得展开：

- 完整规则解释；
- 全部读取文件清单；
- 每一步执行过程；
- 未修改文件长列表；
- 未创建临时文件的长篇说明；
- 详细链路检查过程。

## 11. Blocker 处理

发生 blocker 必须立即停止，不得绕过，并按既有规则写入 `<ERROR_LOG_TEMPLATE>`。

输出格式：

```markdown
## Blocker
- 问题：
- 停止位置：
- 已完成：
- 未执行：
- 建议处理：
```

Blocker 包括但不限于：

- 输入路径不存在、不可读或超出授权范围；
- 需要修改原始资料；
- 需要移动、删除、重命名或覆盖原始资料；
- 目标正式目录无法判断；
- 目标文件同名冲突；
- 资料版本边界不清且影响结论；
- 需要扫描未授权目录；
- 需要写临时脚本或无必要临时文件；
- 命令失败且影响范围无法确认；
- 需要绕过本规则安全边界。

## 12. 外接硬盘资料边界

路径位于 `/Volumes/...` 等外接硬盘时：

- 原文件只读处理，不移动、不删除、不重命名、不覆盖、不修改；
- 只读取用户明确提供的文件或目录，不扩大扫描；
- 草稿写入 `<CODEX_OUTPUT_DIR>`；
- 确认迁入正式库时，只迁移本次生成的 Markdown 成果，不迁移外接硬盘原文件；
- 来源路径保留外接硬盘原路径；

## 13. 质量优先级

1. 按原始资料结构吸收资料主体内容；
2. 判断正式归档路径和上层 MOC；
3. 提炼稳定、单一、可复用且有深度的知识卡片；
4. 建立准确双链和标签；
5. 标注不确定事项。
