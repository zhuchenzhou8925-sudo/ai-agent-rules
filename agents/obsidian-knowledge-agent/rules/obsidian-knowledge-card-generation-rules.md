# Obsidian 知识卡片生成规则

## 1. 规则定位

知识卡片用于沉淀可复用、可追踪、可独立引用的单一知识单元。入口流程、确认节点和短报告格式以 `Obsidian-智能归档入口规则.md` 为准。

知识卡片不替代项目笔记，不替代 MOC，不承担资料索引功能。知识卡片必须从“浅层概念卡片”升级为“原子化深度知识卡片”：每张卡片只讲一个明确知识点，但必须把该知识点讲透。

## 2. 适用场景

适合生成知识卡片：

- 稳定概念、方法、判断标准、能力模型、流程、指标、话术；
- 可被方案、可研、汇报、招采、客户沟通复用的观点；
- 来源明确，可追踪到文件、项目笔记、资料索引或用户输入；
- 能独立说明一个知识点。

不适合：

- 完整项目背景、阶段性推进路径、临时任务；
- 大段原文摘录；
- 来源不明、版本不清、待确认事实；
- 目录页、导航页、资料清单；
- 一个卡片同时讲多个并列知识点。

## 3. 生成数量

完整闭环任务中，首轮默认小批量生成 3-5 张。必须优先保证质量，不为凑数量拆分低质量卡片。

每张卡片只表达一个知识点；同一知识点不得重复生成多个近似卡片。宁可少生成，也不得生成浅层、空泛、只改写标题的卡片。

## 4. 输出路径与命名

草稿默认写入：

```text
<CODEX_OUTPUT_DIR>
```

命名：

```text
知识点名称-知识卡片-YYYYMMDD.md
```

知识点名称应可检索、可复用，不直接使用外部资料原文件名。

```yaml
  - 知识卡片

  human_checked: false
  migration_suggested: false

```

过程信息如必须保留，只能放入 frontmatter 的 `review`、`process` 字段，或放入单独 `_process` 元数据区；不得进入正文。过程信息包括人工检查状态、迁移建议、迁移目录、原文件是否移动、处理状态、草稿状态、执行日志、等待用户确认等。

## 6. 标准结构

知识卡片正文必须使用以下结构，章节为空时保留标题并写明“来源资料未提供明确内容”，不得用过程信息填充：

```markdown
# 知识点标题

## 1. 核心定义
## 2. 问题背景
## 3. 作用机制
## 4. 组成要素
## 5. 实施路径
## 6. 适用场景
## 7. 判断标准
## 8. 适用边界
## 9. 可复用表达
## 10. 来源
```

## 7. 深度要求

每张卡片至少包含：

- 核心定义；
- 问题背景；
- 作用机制；
- 组成要素；
- 实施路径；
- 适用场景；
- 适用边界；
- 可复用表达；
- 来源。

方法论类卡片必须展开：

- 步骤；
- 前置条件；
- 判断标准；
- 落地风险；
- 可迁移和不可迁移的场景。

技术类卡片必须展开：

- 技术作用；
- 数据关系；
- 系统关系；
- 集成逻辑；
- 应用场景；
- 技术边界。

项目案例类卡片必须展开：

- 项目背景；
- 建设内容；
- 实施路径；
- 应用效果；
- 可迁移经验；
- 需要核验的数据或边界。

## 8. 知识与过程隔离

正文只写可复用知识，不写处理过程。

正文禁止写入：

- 是否已人工检查；
- 是否建议迁移；
- 是否等待用户确认；
- 原文件是否移动或未移动；
- 迁移目录建议；
- 处理状态、草稿状态、执行日志；
- Codex 执行过程、读取过程、检查过程。

需要反向打开原文件时，在“来源”章节保留简洁来源信息和可点击原文件链接。

## 9. 双链与标签

- 必须反向链接来源项目笔记、资料索引或 MOC；
- 完整闭环任务中，应链接本次 MOC 和项目笔记；
- 标签至少包含 `#知识卡片`、主题领域标签、状态标签；
- 标签建议如需保留，应放入 frontmatter 或来源章节的简洁关联信息，不得写成正文末尾的过程性清单。

## 10. 禁止事项

- 禁止无来源生成知识卡片；
- 禁止把项目笔记全文拆成大量卡片；
- 禁止将原文大段复制为知识卡片；
- 禁止把候选版本或待确认事实写成稳定知识；
- 禁止自动新建重复主题卡片；
- 禁止未确认迁入正式库；
- 禁止修改原始资料；
- 禁止把迁移建议、处理状态、人工检查状态写入正文；
- 禁止用“定义 + 场景 + 示例表达”替代深度卡片结构。

## 11. 报告

不单独输出知识卡片长报告。由入口规则统一使用短报告。短报告可写迁移、状态、确认事项；这些信息不得写入知识卡片正文。
