# Obsidian 项目笔记生成规则

## 1. 规则定位

本规则用于将项目、事项、资料包或主题材料整理为 Obsidian 项目笔记或资料笔记。入口流程、确认节点和短报告格式以 `Obsidian-智能归档入口规则.md` 为准。

项目笔记的目标是承载资料主线和资料主体内容。项目笔记不是摘要型笔记，必须升级为“全面资料笔记”：读完项目笔记后，应基本理解原始资料的主要内容，只有核图、核数、引用原文、查看版式或查验附件时才需要回看原文件。

## 2. 适用场景

适合生成项目笔记或资料笔记：

- 项目资料、方案、汇报、可研、会议纪要、客户需求；
- BIM、数字孪生、智慧工地、建筑业数字化、AI 项目管理等主题资料；
- 一个资料来源包含背景、目标、任务、技术路线、系统能力、交付成果、案例实践、风险或后续动作；
- 后续可能拆分知识卡片、挂接 MOC、生成方案或招采材料。

不适合：

- 单一概念、方法、判断标准：优先生成知识卡片；
- 纯目录、纯索引：优先生成 MOC 或资料索引；
- 低价值、重复、临时、来源不明资料：只给处理建议。

## 3. 输入要求

必须明确：

- 来源类型和来源路径；
- 是否为正式资料或测试资料；
- 资料主线、项目名称或主题名称；
- 原始资料结构、页数、章节或主要版块；
- 是否存在待确认内容。

来源不足时，在笔记中标注“待确认”，不得自行补全事实。

## 4. 输出路径

草稿默认写入：

```text
<CODEX_OUTPUT_DIR>
```

未经用户确认，不得写入正式知识库目录。用户明确要求重构正式库既有文件时，可在确认目标文件后直接修改指定文件，不移动、删除、重命名原始资料。

## 5. 命名

```text
项目名称-项目笔记-YYYYMMDD.md
主题名称-资料笔记-YYYYMMDD.md
```

文件名应使用稳定主题，不直接照搬外部资料原文件名。

## 6. frontmatter

```yaml
---
title:
type: project_note
status: draft
source:
source_path:
created:
updated:
tags:
  - 项目笔记
links:
  -
review:
  human_checked: false
  migration_suggested: false
  migration_confirmed: false

```

测试输出必须保留 `test.is_test_output: true`。

过程信息如必须保留，只能放入 frontmatter 的 `review`、`process` 字段，或放入单独 `_process` 元数据区；不得进入正文。过程信息包括人工检查状态、迁移建议、迁移目录、原文件是否移动、处理状态、草稿状态、执行日志、等待用户确认等。

## 7. 标准结构

项目笔记正文必须使用以下结构，章节为空时保留标题并写明“原始资料未提供明确内容”，不得用过程信息填充：

```markdown
# 资料标题｜资料笔记

## 1. 资料定位
## 2. 原始资料结构
## 3. 背景与问题
## 4. 核心观点
## 5. 主要内容详解
## 6. 项目案例 / 应用实践
## 7. 技术路径 / 产品路径
## 8. 数据体系 / 标准体系
## 9. 应用场景
## 10. 创新探索
## 11. 可复用观点
## 12. 可复用表达
## 13. 可拆分知识卡片
## 14. 分析备注
## 15. 来源
```

## 8. 内容深度要求

- 必须尽量完整吸收原始文件内容，不得只写几段概括性总结；
- 必须按原始资料结构展开，保留章节、页码、版块、目录线索或材料自身叙事顺序；
- 必须保留关键事实、项目背景、技术路径、系统能力、场景描述、案例数据、产品能力、方法论、实施路径和边界条件；
- 汇报型 PDF 可按页组或版块整理，图片页只能提炼可确认信息，不能编造图中不可识别内容；
- 对图示、表格、案例数字、获奖资质、效率提升等信息，应标注“来自原始资料，正式引用前建议核验”；
- 项目笔记应能支持后续方案、可研、汇报、招投标和客户沟通复用；
- 不把项目阶段性事实直接泛化为通用规律。

## 9. 知识与过程隔离

正文只写对后续理解、复用、写方案、写可研、写投标有价值的知识内容。

正文禁止写入：

- 是否已人工检查；
- 是否建议迁移；
- 是否等待用户确认；
- 原文件是否移动或未移动；
- 迁移目录建议；
- 处理状态、草稿状态、执行日志；
- Codex 执行过程、读取过程、检查过程。

需要反向打开原文件时，在“来源”章节保留简洁来源信息和可点击原文件链接。

## 10. 分析备注

分析判断可以保留，但必须放入“分析备注”章节，不得混入知识定义和事实描述。

分析备注可包含：

- 该资料最可复用的部分；
- 该资料适合挂接到哪个主题；
- 该资料的适用边界；
- 哪些数据需要二次核验；
- 哪些内容适合继续拆分知识卡片。

## 11. 双链与标签

- 关联 MOC、资料索引和知识卡片时使用 `[[页面名]]`。
- 标签至少包含 `#项目笔记` 或 `#资料笔记`、主题领域标签、状态标签。
- 完整闭环任务中，项目笔记应链接本次生成的 MOC 和知识卡片。
- 标签建议如需保留，应放入 frontmatter 或分析备注，不得作为正文末尾的过程性清单。

## 12. 禁止事项

- 禁止修改原始资料；
- 禁止无来源生成项目笔记；
- 禁止复制原文大段堆砌；
- 禁止写成完整汇报稿；
- 禁止未确认迁入正式库；
- 禁止将待确认内容写成确定事实；
- 禁止把迁移建议、处理状态、人工检查状态写入正文；
- 禁止以“摘要、主线、关键内容”替代全面资料笔记结构。

## 13. 报告

不单独输出项目笔记长报告。由入口规则统一使用短报告。短报告可写迁移、状态、确认事项；这些信息不得写入项目笔记正文。
