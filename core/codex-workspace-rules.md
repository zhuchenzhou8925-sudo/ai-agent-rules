## 2. 基本执行规则

- 默认先理解工作目标，再判断任务类型。
- 涉及本地文件时，先扫描、建清单、给建议，不直接移动、删除或重命名。
- 每次执行任务前必须读取 `<ERROR_LOG_TEMPLATE>`。
- 禁止自行绕过 blocker。
- 遇到路径不存在、权限不足、命令失败、文件已存在、文件冲突、范围不明确时，必须停止并报告。
- 不得自行扩大读取、写入、修改范围。
- 未经用户确认，不得直接修改正式知识库目录中的既有内容。
- 用户已明确授权且风险较低的连续操作，可以在授权范围内连续完成；涉及高风险动作、敏感内容、权限问题或 blocker 时必须立即暂停。
- GitHub 同步只允许上传脱敏规则、模板、说明和占位示例，不得上传真实本地绝对路径、真实日志、客户或项目资料、合同、报价、招采资料、Token、密钥或私钥。
- `commit`、`push`、创建 PR、`merge` 必须另行获得用户确认；Codex 不得自行 `merge`，不得执行 `force push`。

## 3. 任务目标驱动与能力自补齐原则

Codex 执行任务时，必须以用户最终想要获得的成果为核心，先判断目标交付物，再选择资料读取、外部检索、数据处理、图表设计、文件生成和自检路径。不得只根据用户已提供资料或知识库已有内容机械输出。

### 3.1 最终交付物优先

执行前必须判断用户最终需要的交付物类型，包括但不限于：

- 知识库入库；
- 方案报告；
- PPT；
- Word 文档；
- Excel 表；
- 数据分析；
- 图表；
- 图片；
- 视频脚本；
- 代码；
- 批量文件处理结果；
- 项目资料整理成果。

### 3.2 知识库不足时主动补齐

当知识库或用户提供资料不足以支撑高质量成果时，不得简单写“资料不足”“建议补充资料”“需人工提供”。应优先自行尝试补齐：

- 外部政策检索；
- 行业资料检索；
- 标准规范检索；
- 政府官网资料检索；
- 公开案例检索；
- 公开数据检索；
- 产品资料读取；
- 本地文件扫描；
- 数据清洗；
- 图表、图片和示意图的需求判断、占位说明和提示词设计；
- 代码处理；
- 成品文件生成。

只有涉及业务口径、敏感案例、正式承诺、对外发布、产品最新状态无法判断时，才留给用户确认。

### 3.3 外部资料来源优先级

外部检索应优先使用可信来源：

1. 国家部委官网；
2. 省级政府官网；
3. 住建厅、发改委、工信部门官网；
4. 地市州政府或主管部门官网；
5. 正式标准发布机构；
6. 权威行业协会；
7. 企业官网或正式产品资料；
8. 主流权威媒体报道；
9. 其他来源仅作为线索，不作为正式依据。

不得优先使用自媒体、营销号、来源不明转载内容。

### 3.4 Word / PPT / Markdown 文档中的图片生成边界规则

在基于知识库输出 Word、PPT、Markdown、汇报材料、方案文档等成果时，Codex 不执行图片生成、AI 作图、视觉渲染、封面图生成、架构图绘制等图片生成类工作。

Codex 只负责完成：

1. 判断文档中是否需要图形、图片、架构图、流程图、封面图、场景图、信息图、图标组合、页面原型示意、数据图表等非文字内容；
2. 在文档对应位置保留清晰的图片或图形占位符；
3. 说明该图片或图形的使用位置、表达目的、核心内容、版式结构、风格要求；
4. 输出可复制给其他 AI 图像工具使用的图片生成提示词；
5. 继续完成文字内容、结构排版和文档输出。

图片实际生成由其他 AI 图像智能体完成，例如 ChatGPT Image、image2、Midjourney、即梦、通义万相等。

Codex 不得因为无法生成图片而中断 Word、PPT、Markdown 或汇报材料输出。涉及智能建造、BIM、AI Agent、数字孪生、智慧工地等领域的图形化表达需求时，同样只输出图形占位、说明和提示词。

如文档需要图片，应使用以下占位格式：

```markdown
【图片 / 图形占位符：名称】

使用位置：
说明该图片用于第几页、第几章或哪个内容模块。

表达目的：
说明该图片要解决什么表达问题。

核心内容：
列出画面中需要呈现的主要元素。

推荐版式：
说明采用横向架构图、纵向流程图、分层结构图、关系图、场景插画、封面视觉图等形式。

风格要求：
说明配色、风格、比例、是否适合 PPT、是否偏正式汇报等。

AI 图片生成提示词：
提供一段可直接复制给其他 AI 图像工具的完整提示词。
```

### 3.5 代码和自动化处理规则

当任务涉及批量处理、数据统计、格式转换、图形占位说明、文件扫描、表格清洗或重复操作时，应优先使用已有工具、一次性命令或内联命令完成。只有在任务重复、规则稳定、输入输出明确且用户授权保留时，才创建长期固定脚本；禁止落盘散乱临时脚本。可处理的场景包括但不限于：

- 扫描目录；
- 生成文件清单；
- 统计 Markdown、Word、PDF、Excel 文件；
- 清洗 Excel 或 CSV；
- 生成数据表、图形占位说明和 AI 图片生成提示词；
- 批量生成 Markdown；
- 批量检查链接；
- 批量检查结构缺项；
- 批量生成 Word、PPT、PDF；
- 生成对照表和台账。

脚本创建和写入仍必须遵守本文件的脚本规则、写入权限规则和关键操作确认规则。

### 3.6 正式输出前自检

正式输出前必须自行检查：

- 内容是否足够深入；
- 观点是否鲜明；
- 政策依据是否充分；
- 数据和案例是否可信；
- 是否存在空泛表达；
- 是否需要图表、图片或示意图占位但未说明；
- 是否需要外部资料但未补充；
- 是否存在未核验却写成确定事实的内容；
- 是否符合目标交付物格式；
- 是否能直接用于用户后续工作。

发现不足时，应自行补强后再输出。

### 3.7 不得把资料缺口写成正文缺陷

正式报告、方案、PPT 正文中，不应把“需补充资料”“需进一步核验事项”作为正文结尾。资料缺口应处理为内部资料检索任务、附录、资料依据清单、内部待确认事项或后续工作台账。正式正文应保持完整，不应让读者感觉报告尚未准备好。

### 3.8 用户确认边界

以下事项必须交由用户确认：

- 是否引用敏感或未公开案例；
- 是否采用某个正式业务口径；
- 是否承诺具体指标；
- 是否确认产品模块最新状态；
- 是否确认商务合作模式；
- 是否对外发送、发布或提交；
- 是否采用某个正式版本作为最终依据。

除此之外，Codex 应尽量自行完成资料补齐、研究、图形化表达需求判断、代码处理和成果生成。

正式任务完成后，应参考 [[任务目标驱动模式验收清单]] 进行结果自检。

## 8. 任务自动路由规则

为降低用户重复输入规则名称的成本，Codex 在每次执行任务前必须先判断用户意图，并自动读取对应规则文件。

### 8.1 Obsidian 知识库任务自动识别

当用户输入中出现以下任一意图时，应自动判断为 Obsidian 知识库相关任务：

- 归档文件
- 归档目录
- 分析文件
- 分析目录
- 整理资料
- 知识化整理
- 入库
- 生成资料索引
- 生成项目笔记
- 生成知识卡片
- 生成 MOC
- 处理知识库资料
- 对文件进行分析和归档
- 对目录进行分析和归档

除非用户明确说明“不要按 Obsidian 规则执行”，否则应按 Obsidian 入口规则执行。

Obsidian 任务默认使用 **Obsidian 完整闭环低消耗模式**：成果标准保持完整闭环，执行过程压缩，草稿先写入 `<CODEX_OUTPUT_DIR>`，用户确认后再一次性迁入正式库。

### 8.2 默认入口规则

当用户只给出文件路径或目录路径，并使用“归档 / 整理 / 知识化 / 入库 / 分析并归档 / 自动闭环”等表述时，默认读取并执行：

```text
<OBSIDIAN_AGENT_RULES_DIR>/Obsidian-智能归档入口规则.md
```

同时根据该入口规则只读取必要的专项规则，不得每次无差别读取全部 Obsidian 规则。

### 8.3 快捷指令解释

以下用户输入应自动解释为对应任务。

#### 8.3.1 归档

用户输入：

```text
归档：/path/to/file-or-folder
```

解释为：

按 `Obsidian-智能归档入口规则.md` 对该路径执行完整闭环低消耗模式：先生成完整草稿和迁移建议，确认后再迁入正式库。

#### 8.3.2 分析

用户输入：

```text
分析：/path/to/file-or-folder
```

解释为：

按 `Obsidian-智能归档入口规则.md` 对该路径执行只读分析，不生成正式成果，不迁移正式目录。

#### 8.3.3 整理

用户输入：

```text
整理：/path/to/file-or-folder
```

解释为：

按 `Obsidian-智能归档入口规则.md` 对该路径执行知识化整理，生成 `<CODEX_OUTPUT_DIR>` 草稿，不自动迁移正式目录。

#### 8.3.4 入库

用户输入：

```text
入库：/path/to/file-or-folder
```

解释为：

按 `Obsidian-智能归档入口规则.md` 执行完整闭环低消耗模式：先完成只读分析和完整草稿成果，输出迁移目录与上层 MOC 挂接建议；用户确认后一次性完成正式迁移、`reviewed` 状态、挂接、统一标签、草稿清理和边界检查。遇到同名文件、目录不明确、版本边界不清、需要覆盖 / 删除 / 重命名正式文件，必须暂停。

### 8.4 具体生成任务路由

如用户明确要求生成具体成果，应自动读取对应规则：

- 资料索引：读取 `Obsidian-资料索引生成规则.md`
- 项目笔记：读取 `Obsidian-项目笔记生成规则.md`
- MOC：读取 `Obsidian-MOC生成规则.md`
- 知识卡片：读取 `Obsidian-知识卡片生成规则.md`
- 资料组处理流程：读取 `Obsidian-资料组知识化处理流程.md`
- 自动归档入口：读取 `Obsidian-智能归档入口规则.md`
- 正式迁移或目录落位：再读取 `directory-map.md`

读取规则以“必要最小”为原则：`CODEX.md`、`<ERROR_LOG_TEMPLATE>`、入口规则和直接相关专项规则即可。样板文件只在需要复用样板时读取全文。

### 8.5 自动执行边界

即使触发自动路由，仍必须遵守以下边界：

- 每次执行前必须读取 `<ERROR_LOG_TEMPLATE>`
- 不得写临时脚本
- 不得创建临时文件
- 不得绕过 blocker
- 不得修改原始资料
- 不得在未确认情况下迁入正式库
- 不得覆盖、删除、重命名正式文件
- 不得自动改名规避同名冲突
- 不得扫描用户未授权目录
- 如正式目录不明确，必须暂停
- 如目标目录存在同名文件，必须暂停
- 如需要修改 `CODEX.md`、`obsidian-write-rules.md`、`directory-map.md`，必须有用户明确授权
