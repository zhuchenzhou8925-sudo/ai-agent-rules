# 任务目标驱动模式验收清单

## 1. 清单用途

本清单用于检查智能体是否围绕最终交付目标，完成资料、数据、图表、图片、文件、格式和质量补齐。它用于任务完成后的质量验收，不替代 `CODEX.md`、Obsidian 写入规则、智能归档入口规则或知识库总规则。

适用任务包括但不限于：

- 报告、方案、可研、汇报材料；
- PPT、Word、Excel、PDF；
- 知识库入库、资料整理、项目资料治理；
- 数据分析、图表生成、图片或示意图生成；
- 文件扫描、批量处理、格式转换和自动化脚本任务。

## 2. 任务目标判断

- [ ] 是否识别最终交付物类型；
- [ ] 是否区分知识库治理、正式报告、PPT、Word、Excel、图片、代码、数据分析等不同任务；
- [ ] 是否明确任务是“整理资料”还是“生成可用成果”；
- [ ] 是否避免只根据已有资料机械输出；
- [ ] 是否明确用户只需要判断方向、业务口径和正式确认事项。

## 3. 本地资料读取检查

- [ ] 是否读取了相关 MOC；
- [ ] 是否读取了相关项目笔记；
- [ ] 是否读取了相关知识卡片；
- [ ] 是否读取了相关资料索引；
- [ ] 是否检查已有正式资料或授权过程记录中的待确认事项；
- [ ] 是否避免遗漏与主题强相关的已入库资料；
- [ ] 是否记录了本地资料不足之处。

## 4. 外部资料补齐检查

- [ ] 知识库不足时，是否主动检索外部资料；
- [ ] 是否优先检索政府官网、主管部门、标准机构、权威协会等来源；
- [ ] 是否补充政策、数据、案例、标准或行业背景；
- [ ] 是否记录来源标题、发布单位、发布时间和链接；
- [ ] 是否避免使用来源不明的自媒体或营销文章作为正式依据；
- [ ] 是否区分已确认事实和仅作线索的信息。

## 5. 数据与证据检查

- [ ] 是否需要数据支撑；
- [ ] 如果需要，是否主动查找或整理数据；
- [ ] 是否对关键数据进行来源核验；
- [ ] 是否避免编造具体数字；
- [ ] 是否对口径不一致的数据作出说明；
- [ ] 是否把未确认数据转为内部待确认事项，而不是写入正文结论。

## 6. 图表与图片检查

- [ ] 是否判断任务需要图表或结构图；
- [ ] 是否应生成政策时间线、推广路径图、平台关系图、成果包结构图、协同机制图、能力矩阵等；
- [ ] 是否主动设计图表内容，而不是等待用户单独要求；
- [ ] 图表是否服务正文表达，而不是装饰；
- [ ] 图表是否有清晰标题、层级和逻辑；
- [ ] 如需图片或示意图，是否主动生成或提出生成方案；
- [ ] 如果需要 PPT，是否考虑每页图文结构。

## 7. 代码与自动化处理检查

- [ ] 是否存在批量文件扫描、数据清洗、表格处理、图表生成、格式转换等需求；
- [ ] 是否应主动编写脚本完成；
- [ ] 是否避免把可自动完成的重复工作交给用户；
- [ ] 是否输出了可追溯的处理结果；
- [ ] 是否保护原始文件，避免误删、误改、误移动。

## 8. KnowledgeBase 写入边界检查

- [ ] 验证产物是否未写入 KnowledgeBase；
- [ ] 临时报告是否输出到 `<CODEX_OUTPUT_DIR>`；
- [ ] 运行日志是否输出到 `<CODEX_CONTROL_DIR>/<RUN_LOG_DIR>`；
- [ ] 治理备份是否输出到 `<CODEX_CONTROL_DIR>/<GOVERNANCE_ARCHIVE_DIR>`；
- [ ] KnowledgeBase 是否只保留用户确认后的正式成果；
- [ ] 是否不存在未经确认的过程目录；
- [ ] 图表清单、外部资料准备清单、内部待确认事项和中间输出是否未写入 KnowledgeBase。

## 9. 输出成果完整性检查

按交付物类型检查。

### 9.1 报告 / 方案

- [ ] 是否有明确判断；
- [ ] 是否有政策和行业依据；
- [ ] 是否有问题分析；
- [ ] 是否有实施路径；
- [ ] 是否有分工机制；
- [ ] 是否有风险边界；
- [ ] 是否有近期行动；
- [ ] 是否避免空泛表述；
- [ ] 是否删除“需补资料”式正文结尾；
- [ ] 是否把内部待确认事项放入附录或台账。

### 9.2 PPT

- [ ] 是否有清晰汇报主线；
- [ ] 是否每页一个重点；
- [ ] 是否设计图表和结构图；
- [ ] 是否符合模板要求；
- [ ] 是否检查字体、字号、页码、logo、底部色块；
- [ ] 是否避免满屏文字；
- [ ] 是否适合直接汇报。

### 9.3 Word

- [ ] 是否结构完整；
- [ ] 是否标题层级清晰；
- [ ] 是否适合正式提交；
- [ ] 是否有必要的表格、图示和附录；
- [ ] 是否避免口语化和草稿痕迹。

### 9.4 Excel / 表格

- [ ] 是否字段完整；
- [ ] 是否格式规范；
- [ ] 是否可筛选、可统计、可复用；
- [ ] 是否保留来源和备注；
- [ ] 是否避免把说明文字混入数据列。

### 9.5 知识库入库

- [ ] 是否完成 MOC 挂接；
- [ ] 是否生成项目笔记；
- [ ] 是否生成知识卡片；
- [ ] 是否更新资料索引；
- [ ] 是否正文无流程污染；
- [ ] 是否治理事项进入 `<CODEX_CONTROL_DIR>/<RUN_LOG_DIR>` 或治理归档目录；
- [ ] 是否内部链接无断链。

## 10. 正式输出前自检

- [ ] 内容是否足够深入；
- [ ] 观点是否鲜明；
- [ ] 结构是否完整；
- [ ] 依据是否充分；
- [ ] 图表是否必要且已生成；
- [ ] 是否有数据或案例支撑；
- [ ] 是否存在未核验却写死的内容；
- [ ] 是否存在治理过程或内部待办进入正文；
- [ ] 是否存在验证产物、过程文件或临时资料写入 KnowledgeBase；
- [ ] 是否满足用户最终交付目标；
- [ ] 是否可以直接进入用户判断或修改阶段。

## 11. 用户确认边界检查

确认是否只把以下事项留给用户：

- [ ] 业务口径；
- [ ] 敏感案例；
- [ ] 未公开资料；
- [ ] 产品最新状态；
- [ ] 商务合作方式；
- [ ] 具体指标承诺；
- [ ] 正式版本确认；
- [ ] 是否对外提交、发布或发送。

除此之外，智能体应尽量自行完成资料补齐、检索、图表、代码、文件生成和质量自检。

## 12. 验收结论模板

每次任务完成后，可按以下格式输出：

```markdown
## 任务目标驱动验收结论
- 最终交付物是否明确：
- 本地知识库是否已读取：
- 外部资料是否已补充：
- 政策 / 数据 / 案例依据是否充分：
- 是否生成必要图表 / 图片 / 表格：
- 是否使用代码或工具完成可自动化处理：
- 正文是否存在资料缺口式表达：
- 是否存在未核验事实写死：
- 验证产物是否未写入 KnowledgeBase：
- 临时报告是否输出到工作台级 <CODEX_OUTPUT_DIR>：
- 运行日志是否输出到 Codex_Control/<RUN_LOG_DIR>：
- 治理备份是否输出到 <GOVERNANCE_ARCHIVE_DIR>：
- KnowledgeBase 是否只保留正式成果：
- 是否存在未经确认的过程目录：
- 是否符合目标格式：
- 是否需要用户确认：
- 结论：通过 / 部分通过 / 不通过
```
