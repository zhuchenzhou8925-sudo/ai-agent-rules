# GitHub 工作流规则

## 基础规则
1. 不得上传敏感文件。
2. 不得直接 push 到 main，除非是首次初始化且用户明确确认。
3. 后续修改优先进入 `draft/*` 分支。
4. push 前必须输出上传前确认报告。
5. PR 前必须输出确认报告。
6. 合并 main 前必须等待用户确认。
7. 如发现疑似敏感文件，必须停止。
8. 对非技术用户，Codex 必须用中文解释修改内容、影响范围、风险等级和回滚方式。
9. 工具类脚本默认应支持 dry-run。
10. 不得修改现有本机整体工作区结构。
11. commit、push、PR、merge 必须另行确认；Codex 不得自行 merge。
12. 禁止 force push。
13. GitHub 同步必须使用脱敏内容，不得上传真实本地绝对路径、真实日志、客户或项目资料、合同、报价、招采资料、Token、密钥或私钥。

## 分支规则
- `main`：稳定版本，只存放已确认内容。
- `draft/*`：Codex 修改草稿分支。
- `backup/*`：重大调整前备份分支。

## 后续默认规则
1. 首次初始化后，不得直接修改并 push 到 `main`。
2. Codex 后续所有实质性修改，默认新建 `draft/YYYY-MM-DD-task-name` 分支。
3. 小型文档错字修正可以建议直接 commit，但 push 前仍需用户确认。
4. 涉及规则、脚本、目录结构、自动化工具、删除文件、大规模重命名的修改，必须走 `draft/*` 分支。
5. 合并到 `main` 前必须由用户确认。
6. 低风险且已明确授权的本地修改可以连续完成；commit、push、PR、merge 仍必须在下一个确认节点等待用户确认。

## 修改类型自动判断规则
Codex 必须自行判断修改类型，不得要求用户判断“小修改 / 中等修改 / 大修改”。判断时必须结合任务目标、涉及文件数量、规则逻辑影响、目录结构影响、脚本或自动化影响、删除或批量处理风险、敏感信息风险，并用中文说明依据。

| 修改类型 | 判断标准 | 推荐流程 |
|---|---|---|
| 小修改 | 文档错字、README 小补充、格式/标题/表格调整；不改变规则逻辑、不新增脚本、不删除文件、不调整目录结构、不影响 Codex 执行行为。 | 检查 -> 本地修改 -> 自检 -> commit 前确认 -> commit -> push 前确认 -> 用户确认后 push |
| 中等修改 | 补充规则文件、调整提示词、新增模板、修改 Codex 工作流、修改 Obsidian 智能体规则、增加清单；或修改超过 2 个文件但不涉及删除、脚本、目录结构。 | 建议使用 `draft/*` 分支 -> 修改 -> 自检 -> commit 前确认 -> commit -> push 前确认 -> push draft -> PR 前确认 -> 创建 PR -> 用户确认后合并 |
| 大修改 | 修改核心规则、目录结构、新增/删除/重命名多个文件、新增或修改脚本/自动化工具、批量处理、复制/迁移/归档、跨仓库同步、GitHub 工作流规则变更，或可能影响 Codex 后续行为。 | 必须使用 `draft/*` 分支 -> 修改 -> 自检 -> 本地测试 -> commit 前确认 -> commit -> push 前确认 -> push draft -> PR 前确认 -> 创建 PR -> 用户确认后合并 |
| 高风险修改 | 涉及密钥、凭据、真实业务资料、真实日志、删除/覆盖/批量重命名、权限、认证、部署、CI/CD、force push 或不可逆操作。 | 暂停执行 -> 输出风险、影响范围和建议 -> 等待用户确认 |
| 禁止执行 | 涉及 `.env`、API Key、Token、password、secret、credential、private key、SSH 私钥、证书、客户资料、合同、报价、招采文件、内部资料、未脱敏业务资料、`_Codex_Output/` 未筛选内容、真实日志。 | 停止执行并说明原因 |

## 风险等级自动判断规则
| 风险等级 | 判断标准 | 处理要求 |
|---|---|---|
| 低 | 文档小改、说明补充、格式整理；不影响目录结构和执行逻辑。 | 可以建议继续执行；push 前仍需用户确认。 |
| 中 | 规则、提示词、模板、多个文件变更；影响使用方式但不涉及脚本、删除或敏感资料。 | 可继续，但必须输出确认报告；优先建议 `draft/*` 分支。 |
| 高 | 脚本、目录结构、删除、批量处理、跨仓库同步、自动化工具、GitHub 工作流变化。 | 必须使用 `draft/*` 分支，并在关键节点等待用户确认。 |
| 禁止 | 密钥、凭据、真实业务资料、真实日志、未脱敏资料、force push 或不可逆高危动作。 | 停止执行，不得绕过。 |

Codex 必须同步说明是否存在 blocker、是否需要用户确认、是否需要 `draft/*` 分支、是否需要 PR、是否需要提交给 ChatGPT 协助判断。

## 自动流程选择规则
1. 低风险小修改：允许建议在 `main` 本地修改，但 commit 和 push 前都必须等待确认。
2. 中等修改：默认建议 `draft/*` 分支；如用户明确要求且 Codex 判断风险低，可说明理由后等待确认再直接处理。
3. 大修改：必须进入 `draft/*` 分支，不得直接 push 到 `main`。
4. 高风险修改：暂停，输出风险说明、影响范围、替代方案和需要确认的动作。
5. 禁止执行项：拒绝执行并说明原因。

|---|---|---|---|
## 3. 敏感信息检查

## PR 合并规则
1. Codex 不得自行合并 PR。
2. 用户确认前，不得 merge 到 `main`。
3. 高风险修改不得自动合并。
4. 删除文件、大规模重命名、权限、部署、密钥、认证相关修改必须单独确认。
5. 合并后必须输出合并后自检报告。
6. 任何情况下不得 force push。
