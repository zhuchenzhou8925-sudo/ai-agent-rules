# GitHub 工作流规则

## 基础规则
1. 不得上传敏感文件。
2. 不得直接 push 到 main，除非是首次初始化且用户明确确认。
3. 后续修改优先进入 `draft/*` 分支。
4. push 前必须输出上传前确认报告。
5. PR 前必须输出确认报告。
6. 合并 main 前必须等待用户确认。
7. 如发现疑似敏感文件，必须停止。
8. 对非技术用户，Codex 必须用中文解释修改内容、影响范围、风险等级和回滚方式。
9. 工具类脚本默认应支持 dry-run。
10. 不得修改现有本机整体工作区结构。

## 分支规则
- `main`：稳定版本，只存放已确认内容。
- `draft/*`：Codex 修改草稿分支。
- `backup/*`：重大调整前备份分支。

## 后续默认规则
1. 首次初始化后，不得直接修改并 push 到 `main`。
2. Codex 后续所有实质性修改，默认新建 `draft/YYYY-MM-DD-task-name` 分支。
3. 小型文档错字修正可以建议直接 commit，但 push 前仍需用户确认。
4. 涉及规则、脚本、目录结构、自动化工具、删除文件、大规模重命名的修改，必须走 `draft/*` 分支。
5. 合并到 `main` 前必须由用户确认。

## Push 前确认报告模板
```markdown
# GitHub push 前确认报告
## 1. 本次任务目标
## 2. 修改文件清单
| 文件 | 操作 | 中文说明 | 风险等级 |
|---|---|---|---|
## 3. 敏感信息检查
## 4. 是否影响原始工作区
## 5. 是否影响 Obsidian 正式库
## 6. 是否建议 push
## 7. 回滚方式
## 8. 需要用户确认的动作
```

## PR 前确认报告模板
```markdown
# Pull Request 创建前确认报告
## 1. PR 目标
## 2. 源分支
## 3. 目标分支
## 4. 修改摘要
## 5. 风险等级
## 6. 敏感信息检查
## 7. 是否建议创建 PR
## 8. PR 标题建议
## 9. PR 描述建议
## 10. 需要用户确认的动作
```

## PR 合并规则
1. Codex 不得自行合并 PR。
2. 用户确认前，不得 merge 到 `main`。
3. 高风险修改不得自动合并。
4. 删除文件、大规模重命名、权限、部署、密钥、认证相关修改必须单独确认。
5. 合并后必须输出合并后自检报告。
