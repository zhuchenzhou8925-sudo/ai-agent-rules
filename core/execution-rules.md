# 通用执行规则

## 目标
规范智能体执行任务时的边界、确认点和交付要求。

## 规则
- 先理解任务目标，再判断任务类型。
- 涉及本地文件时，先扫描和建立清单。
- 涉及修改、复制、上传时，必须先输出确认报告。
- 不确定或高风险内容必须暂停并说明原因。
- 明确授权范围内的低风险任务可以连续执行；高风险任务、权限问题、敏感内容和 blocker 必须暂停。
- 禁止落盘散乱临时脚本；优先使用已有工具、一次性命令或内联命令，只有重复且稳定的授权场景才创建固定脚本。
- 面向非技术用户时，输出应简洁、直接、结构化。

## GitHub 执行边界
- 后续实质性修改默认使用 `draft/*` 分支。
- 未经用户确认，不得直接 push 到 `main`。
- commit、push、PR、merge 均必须另行确认。
- push 前必须输出 GitHub push 前确认报告。
- PR 前必须输出 Pull Request 创建前确认报告。
- 合并 `main` 前必须等待用户确认。
- Codex 不得自行 merge，不得 force push。
- GitHub 同步必须脱敏，不得上传真实本地绝对路径、真实日志、客户或项目资料、合同、报价、招采资料、Token、密钥或私钥。
- 工具类修改必须优先支持 dry-run。

## 执行前自确认
每次修改前，Codex 必须检查：
- 任务目标是否清晰。
- 涉及仓库和文件是否明确。
- 是否影响 `main` 稳定版本。
- 是否需要 `draft/*` 分支。
- 是否需要 PR。
- 是否存在敏感信息风险。
- 是否影响原始工作区。
- 是否影响 Obsidian 正式库。
- 是否涉及删除、覆盖、批量处理或重命名。
- 是否需要 `backup/*` 分支。
- 是否存在 blocker。

执行前必须输出：

```markdown
# Codex 执行前自确认报告
## 1. 我对任务的理解
## 2. 我判断的修改类型
小修改 / 中等修改 / 大修改 / 高风险修改 / 禁止执行
## 3. 判断依据
## 4. 涉及仓库和文件
## 5. 风险等级
低 / 中 / 高 / 禁止
## 6. 是否存在 blocker
## 7. 我建议的执行流程
## 8. 是否需要 draft 分支
## 9. 是否需要 PR
## 10. 是否需要用户确认
## 11. 用户只需确认的事项
请用户只确认：是否同意按上述建议执行。
```

## 执行后自检
修改完成后必须输出：

```markdown
# Codex 执行后自检报告
## 1. 本次实际修改内容
## 2. 修改文件清单
## 3. 是否符合执行前判断
## 4. 敏感信息检查
## 5. 是否影响原始工作区
## 6. 是否影响 Obsidian 正式库
## 7. 是否存在删除、覆盖或批量重命名
## 8. 本地 Git 状态
## 9. 风险等级复核
## 10. 是否建议进入 commit / push / PR
## 11. 需要用户确认的下一步
```

## 不确定问题处理模板
如果 Codex 无法判断技术风险，不得自行猜测。必须生成：

```markdown
# 待 ChatGPT 协助判断的问题包
## 1. 当前任务背景
## 2. 当前目标
## 3. 已发现问题
## 4. 不确定点
## 5. 需要 ChatGPT 判断的问题
## 6. Codex 初步建议
## 7. 已排除的风险
## 8. 可能的执行路径
## 9. 推荐用户确认方式
## 10. 是否暂停执行
暂停执行，等待用户确认。
```
