# 文件安全规则

## 基础规则
1. 不得上传敏感文件。
2. 不得直接 push 到 main，除非是首次初始化且用户明确确认。
3. 后续修改优先进入 `draft/*` 分支。
4. push 前必须输出上传前确认报告。
5. PR 前必须输出确认报告。
6. 合并 main 前必须等待用户确认。
7. 如发现疑似敏感文件，必须停止。
8. 对非技术用户，Codex 必须用中文解释修改内容、影响范围、风险等级和回滚方式。
9. 工具类脚本默认应支持 dry-run。
10. 不得修改现有本机整体工作区结构。
11. GitHub 同步前必须进行脱敏检查，只允许上传规则、模板、说明和脱敏样例。
12. 禁止落盘散乱临时脚本；自动化处理优先使用已有工具、一次性命令或内联命令。

## 禁止内容
- `.env`、API Key、Token、密码、SSH 私钥、证书。
- 客户资料、合同、报价、招采文件。
- 内部资料、原始资料、未脱敏资料。
- Obsidian 正式知识库全文。
- `_Codex_Output/` 未筛选内容。
- 真实本地绝对路径。
- 真实错误日志或运行日志原文。

## 绝对禁止上传
任何情况下都不得上传：
- `.env`
- API Key
- Token
- password
- secret
- credential
- private key
- SSH 私钥
- 证书
- 客户资料
- 合同
- 报价
- 招采文件
- 内部资料
- 未脱敏业务资料
- `_Codex_Output/` 未筛选内容
- `<ERROR_LOG_TEMPLATE>` 对应的真实日志
- 真实本地绝对路径
- 真实项目名称或客户名称

## 命中词解释
如果上述词只出现在 `.gitignore`、禁止上传说明或安全规则中，不算敏感内容本体，但必须在检查报告中说明。

公开仓库中允许使用 `<WORKSPACE_ROOT>`、`<CODEX_CONTROL_DIR>`、`<CODEX_OUTPUT_DIR>`、`<KNOWLEDGE_BASE_DIR>`、`<AI_OUTPUT_DIR>`、`<SKILL_DIR>` 等占位符；不得写入占位符对应的真实路径。

## 自动化工具安全规则
对 `automation-tools` 后续新增脚本或工具必须遵守：
1. 默认支持 dry-run。
2. 默认不直接修改原始文件。
3. 修改前输出影响范围。
4. 修改后输出结果清单。
5. 删除、覆盖、批量重命名前必须单独确认。
6. 工具 README 必须说明运行方法、输入、输出、风险、回滚方式。
