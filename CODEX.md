# Codex 执行入口规则

## 基本原则
- 默认使用中文说明修改内容、影响范围、风险等级和回滚方式。
- 未经用户确认，不得 push、创建 PR 或合并 main。
- 不复制、不上传客户资料、合同、报价、招采文件、内部敏感资料和未脱敏资料。
- 复制现有规则文件前，必须先生成候选清单和映射表。

## 本仓库当前状态
本仓库已初始化基础目录，并包含经过脱敏整理的智能体规则副本。

## GitHub 后续维护规则
- `main` 为稳定版本，只存放用户已确认内容。
- 后续实质性修改默认新建 `draft/YYYY-MM-DD-task-name` 分支。
- 重大调整前可创建 `backup/*` 分支。
- 不得直接 push 到 `main`，除非用户明确确认。
- push 前必须输出 GitHub push 前确认报告。
- 创建 PR 前必须输出 Pull Request 创建前确认报告。
- Codex 不得自行合并 PR，合并 `main` 前必须等待用户确认。
- 工具类修改必须优先支持 dry-run。

## GitHub 维护智能判断规则
- Codex 必须自动判断修改类型：小修改 / 中等修改 / 大修改 / 高风险修改 / 禁止执行。
- Codex 必须自动判断风险等级：低 / 中 / 高 / 禁止，并说明判断依据。
- 用户不需要判断是否使用 `draft/*` 分支、是否创建 PR、是否可以 push；Codex 必须给出中文建议和原因。
- 小修改可建议在 `main` 本地处理，但 commit 和 push 前仍需确认。
- 中等修改默认建议使用 `draft/*` 分支；如风险低且用户明确要求，可说明原因后等待确认再直接处理。
- 大修改必须使用 `draft/*` 分支，并在合并 `main` 前等待用户确认。
- 高风险修改必须暂停；禁止执行项必须拒绝执行并说明原因。

## Codex 自检与自确认
每次执行前必须输出“Codex 执行前自确认报告”，说明任务理解、修改类型、判断依据、涉及仓库和文件、风险等级、blocker、建议流程、是否需要 draft 分支、是否需要 PR、是否需要用户确认。

每次执行后必须输出“Codex 执行后自检报告”，说明实际修改内容、文件清单、敏感信息检查、原始工作区影响、Obsidian 正式库影响、Git 状态、风险复核和下一步建议。

## 用户确认简化
面向非技术用户时，Codex 必须把技术判断转成中文结论。用户只需回复：同意执行 / 暂停 / 改为 draft 分支 / 改为只检查不修改 / 发给 ChatGPT 判断。

## 不确定问题处理
如果 Codex 无法判断技术风险，不得自行猜测。必须生成“待 ChatGPT 协助判断的问题包”，并暂停等待用户确认。
