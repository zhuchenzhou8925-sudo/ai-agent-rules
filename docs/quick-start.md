# Quick Start

本指南说明如何用最小步骤把本仓库中的规则模板接入到本地 Obsidian 知识库和 Codex 或类似本地文件处理智能体中。

## 最小使用流程

### 工作区级入口：Knowledge Management Skill v1

当任务从整个工作区、多个资料目录或知识库治理视角开始时，优先使用：

- `skills/knowledge-management/SKILL.md`

新用户请优先安装 `skills/knowledge-management/`。`skills/obsidian-project-knowledge-skill/` 为历史版或兼容示例，不作为默认安装入口。

该入口先做工作区级只读分析，再决定是否进入 Obsidian / Markdown 正式知识化入档流程。当前版本支持资料类型识别、资料索引、多类型主题笔记、MOC、分型知识卡片、双链、标签、合并去重、版本判断、质量检查和上层 MOC 挂接建议。

测试或验证新版 Skill 时，输出材料应放在规则仓库的测试或报告目录，不写入正式 KnowledgeBase，也不写入 KnowledgeBase 内的 `_Codex_Output`。用户授权资料正式处理时，默认判断如何进行 formal knowledge filing，而不是先生成草稿再迁移。

### 第一步：准备 Obsidian 知识库

先准备一个本地 Obsidian 知识库，建议至少包含项目资料、产品能力、行业场景、商务资料、汇报材料、可复用知识、MOC 和资料索引等区域。

不要一开始就把全部历史资料放入处理范围。建议先选择一个小型、低风险、已授权的资料目录作为测试对象。

### 第二步：准备 Codex 或类似本地文件处理智能体

准备一个能够读取本地文件、生成 Markdown、执行只读扫描和按规则工作的智能体环境。该环境应能遵守以下边界：

- 只读取用户授权范围。
- 文件处理前先生成清单和建议。
- 未经确认不移动、删除、重命名原始文件。
- 未经确认不写入正式知识库目录。

### 第三步：复制规则模板

从本仓库复制需要的规则模板。常用入口包括：

- `skills/knowledge-management/SKILL.md`
- `core/codex-workspace-rules.md`
- `agents/obsidian-knowledge-agent/rules/obsidian-write-rules.md`
- `agents/obsidian-knowledge-agent/rules/obsidian-smart-archive-entry-rules.md`
- `agents/obsidian-knowledge-agent/rules/obsidian-knowledge-base-rules.md`
- `agents/obsidian-knowledge-agent/source-file-mapping.md`

这些文件是脱敏副本，需要根据你的本地路径重新配置。

### 第四步：配置本地路径

把模板中的占位路径替换为你的本地目录，例如：

- `<WORKSPACE_ROOT>`：工作台根目录。
- `<WORKBENCH_ROOT>`：工作台或日常控制区。
- `<RULES_REPO>`：规则源码仓库。
- `<LEGACY_RULES_DIR>`：旧版规则目录，只读参照。
- `<KNOWLEDGE_BASE>`：正式 Obsidian / Markdown 知识库。
- `<SOURCE_ARCHIVE>`：原始资料和最终原始成果归档区。
- `<CODEX_CONTROL_DIR>`：智能体控制规则目录。
- `<CODEX_OUTPUT_DIR>`：仅用于测试、预览、异常报告、过程检查、备份或用户明确要求的临时输出。
- `<OBSIDIAN_AGENT_RULES_DIR>`：Obsidian 知识库规则目录。
- `<RUN_LOG_DIR>`：运行日志目录。

不要把真实客户资料路径、密钥或私有业务内容写入公开仓库。

### 第五步：放入待整理资料

把待整理资料放入一个明确的待处理目录，或提供一个已授权的原始资料目录路径。

第一次使用时，建议选择少量文件进行验证，例如一个项目汇报、一个方案文档或一组已脱敏资料。

### 第六步：让智能体只读扫描

第一次使用必须只读扫描。扫描目标是：

- 对工作区级任务，先扫描一级和二级目录，建立目录职责图。
- 判断路径是否存在、可读、在授权范围内。
- 建立文件清单。
- 判断文件类型、数量、版本和重复关系。
- 识别主要资料类型、次要资料类型、混合资料关系、重复或版本关系。
- 判断建议生成的项目、研究、文献、技术、产品、政策标准、商务、方法或扩展主题笔记类型。
- 输出处理建议。

不要在第一次扫描时直接移动、删除、重命名原始文件。

### 第七步：确认分类方案

人工确认智能体输出的分类方案和处理建议。重点确认：

- 哪些资料应生成哪类主题笔记或资料索引。
- 哪些资料适合生成分型知识卡片。
- 是否需要建立或挂接 MOC。
- 是否需要生成资料索引。
- 哪些资料不应入库或需要脱敏。

用户提供的材料默认有价值。系统的重点不是简单判断资料是否有价值，而是判断如何入库、如何拆解、如何挂接和如何复用。

### 第八步：执行入库 / 生成卡片 / 生成索引

用户授权正式入档且目标目录明确后，再执行具体动作：

- 生成资料索引。
- 生成多类型主题笔记。
- 生成具备复用价值的分型知识卡片。
- 生成或挂接 MOC。
- 建立双链和标签建议。
- 执行合并、去重、版本判断和统一质检。

入库不是只生成知识卡片，也不是草稿迁移流程。完整 formal knowledge filing 应形成可追踪、可挂接、可复用的正式知识结构。

### 第九步：人工复核

正式入档或复用前必须人工复核：

- 来源是否准确。
- 敏感内容是否已脱敏。
- 业务口径是否准确。
- 标签和双链是否合适。
- 是否存在待确认事实。
- 是否需要补充 MOC 或资料索引。

涉及对外提交、正式汇报、商务报价、合同条款或承诺指标时，应由人工最终确认。

## 图片和图形内容

当文档需要架构图、流程图、封面图、场景图、信息图或页面原型示意时，Codex 只输出占位符、说明和 AI 图片生成提示词，不直接生成图片。

图片实际生成应交给 ChatGPT Image、image2、Midjourney、即梦、通义万相或其他图像工具完成。
